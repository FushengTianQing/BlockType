## 一、总体架构设计

### 1.1 多前端多后端架构总览

#### 1.1.1 编译器分层架构图

```
┌─────────────────────────────────────────────────────────────────────┐
│                        CompilerInstance (管线编排器)                  │
│  ┌───────────────────────────────────────────────────────────────┐  │
│  │                  Frontend Registry (前端注册表)                 │  │
│  │  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐     │  │
│  │  │C++ Front │  │C Front   │  │Rust Front│  │  ...     │     │  │
```

#### 1.1.2 各层职责边界

| 层次 | 职责 | 输入 | 输出 | 核心接口 |
|------|------|------|------|----------|
| **前端层** | 源代码 → AST → BTIR | 源代码文本 | `btir::Module` | `FrontendBase::compile()` |
| **IR 中间层** | 类型/值/指令的统一表示 | `btir::Module` | `btir::Module` | `btir::Module`, `btir::Pass` |
| **后端层** | BTIR → 目标代码 | `btir::Module` | 目标文件/汇编 | `BackendBase::emitObject()` |

#### 1.1.3 解耦策略

1. **依赖方向单向化**：`Frontend → BTIR ← Backend`，前端和后端之间零依赖
2. **接口抽象**：前端通过 `btir::Builder` 构建 IR；后端通过遍历 `btir::Module` 消费 IR
3. **类型系统独立**：BTIR 拥有自有的类型系统（`btir::Type`），与 `llvm::Type` 和 `QualType` 均独立
4. **值系统独立**：BTIR 拥有自有的值系统（`btir::Value`），与 `llvm::Value` 独立

#### 1.1.4 自由组合机制

```bash
bt --frontend=cpp --backend=llvm -O2 source.cpp -o source.o
bt --frontend=cpp --backend=cranelift source.cpp -o source.o
bt --frontend=rust --backend=llvm source.rs -o source.o
bt --frontend=cpp --emit-btir source.cpp -o source.btir
bt --backend=llvm source.btir -o source.o
```

```cpp
auto Frontend = FrontendRegistry::instance().create("cpp", Opts);
auto Backend  = BackendRegistry::instance().create("llvm", Opts);
auto IRModule = Frontend->compile(Source);
Backend->emitObject(*IRModule, OutputPath);
```

### 1.2 中间表示（IR）层设计

#### 1.2.1 核心决策：自定义 BTIR vs MLIR vs 其他

| 方案 | 优势 | 劣势 | 适合度 |
|------|------|------|--------|
| **自定义 BTIR** | 完全控制；无外部依赖；轻量；可针对 C++ 语义优化 | 需自建基础设施 | ★★★★★ |
| **MLIR** | 成熟基础设施；Dialect 支持多前端多后端 | 重度依赖 LLVM；C++ 语义映射工作量大 | ★★★☆☆ |
| **复用 LLVM IR** | 零开发量 | 完全绑定 LLVM；违背架构红线 | ★☆☆☆☆ |
| **Cranelift IR** | 轻量；与 Cranelift 天然匹配 | Rust 生态绑定；C++ 语义支持弱 | ★★☆☆☆ |

**最终选择：自定义 BTIR**

理由：(1) 架构红线要求前后端完全解耦；(2) 可精确表达 C++ 特有语义；(3) 可从最小子集渐进扩展；(4) 不预设后端假设；(5) 完全自主可控。

#### 1.2.2 IR 类型系统

```
btir::Type (抽象基类)
├── VoidType
├── IntegerType        (i1/i8/i16/i32/i64/i128)
├── FloatType          (f16/f32/f64/f80/f128)
├── PointerType        (T*)
├── ArrayType          (T[N])
├── StructType         ({name, fields, packed})
├── FunctionType       (ret, params, variadic)
├── VectorType         (T<N>)  SIMD
└── OpaqueType         前向声明
```

- 类型唯一化通过 `BTIRContext` 实现
- 与 `QualType` 的映射：`IRTypeMapper::mapType(QualType) → btir::Type*`（前端适配层）
- 与 `llvm::Type` 的映射：`LLVMTypeMapper::mapType(btir::Type*) → llvm::Type*`（后端适配层）

#### 1.2.3 IR 指令集

SSA 形式，后端无关：

```
终结:   Return / Branch / Switch / Unreachable / Invoke
算术:   BinaryOp(add/sub/mul/udiv/sdiv/urem/srem/shl/shr/and/or/xor) / UnaryOp(neg/not) / Compare
内存:   Alloca / Load / Store / GEP / Memcpy
转换:   Cast(bitcast/zext/sext/trunc/fptoui/uitofp/...) / Select
调用:   Call
聚合:   ExtractValue / InsertValue
元信息: DebugLoc / Metadata
```

C++ 特有语义（vtable/RTTI/dynamic_cast）通过**函数调用+约定**表达，而非特殊指令。

#### 1.2.4 IR 模块层次

```
btir::Module
├── name, target_triple, data_layout
├── functions: [btir::Function { name, linkage, type, params, basic_blocks }]
│   └── btir::BasicBlock { label, instructions }
├── global_variables: [btir::GlobalVariable { name, type, linkage, initializer }]
├── global_aliases
└── metadata
```

#### 1.2.5 IR 与 AST 的映射

| AST 节点 | BTIR 映射 |
|----------|-----------|
| TranslationUnitDecl | btir::Module |
| FunctionDecl | btir::Function |
| VarDecl(全局) | btir::GlobalVariable |
| VarDecl(局部) | AllocaInst + Load/Store |
| CXXRecordDecl | btir::StructType |
| CallExpr | btir::CallInst |
| BinaryOperator | btir::BinaryOpInst |
| IfStmt | btir::BranchInst |
| CXXTryStmt | btir::InvokeInst |

#### 1.2.6 IR 与后端对接

每个后端实现 `IRConsumer` 接口遍历 `btir::Module`：

```cpp
class IRConsumer {
public:
    virtual ~IRConsumer() = default;
    virtual void processModule(const btir::Module &M) = 0;
    virtual void processFunction(const btir::Function &F) = 0;
    virtual void processInstruction(const btir::Instruction &I) = 0;
};
```

#### 1.5.1 CompilerInstance 重构

```cpp
class CompilerInstance {

```

#### 1.5.2 管线流程

```
compileFile("source.cpp"):
  1. 解析选项 → createFrontend("cpp") + createBackend("llvm")
  2. runFrontend → Preprocessor → Parser → Sema → AST → ASTToIRConverter → btir::Module
  3. runIRPipeline → VerifierPass + OptimizationPass(未来)
  4. runBackend → IRToLLVMConverter → llvm::Module → LLVM优化 → 目标文件
  5. runLinker → 链接可执行文件
```

#### 1.5.3 新增编译选项

```
--frontend=<name>     选择前端，默认自动选择
--backend=<name>      选择后端，默认 llvm
--emit-btir           输出 BTIR 文本格式
--emit-btir-bitcode   输出 BTIR 二进制格式
--btir-only           只输出 BTIR，不运行后端
--load-btir=<file>    从 BTIR 文件加载，跳过前端
```

```cpp
// include/blocktype/IR/IRType.h
enum class TypeID { Void, Integer, Float, Pointer, Array, Struct, Function, Vector, Opaque };

class Type {
protected:
    TypeID ID; BTIRContext &Ctx;
public:
    Type(TypeID ID, BTIRContext &Ctx) : ID(ID), Ctx(Ctx) {}
    TypeID getTypeID() const { return ID; }
    virtual uint64_t getSizeInBits() const = 0;
    virtual uint64_t getAlignment() const = 0;
    virtual void print(raw_ostream &OS) const = 0;
    virtual bool equals(const Type *Other) const = 0;
    // 快捷查询
    bool isVoidTy() const { return ID == TypeID::Void; }
    bool isIntegerTy() const { return ID == TypeID::Integer; }
    bool isPointerTy() const { return ID == TypeID::Pointer; }
    bool isStructTy() const { return ID == TypeID::Struct; }
    bool isFunctionTy() const { return ID == TypeID::Function; }
};

class IntegerType : public Type {
    unsigned BitWidth;
public:
    unsigned getBitWidth() const { return BitWidth; }

class PointerType : public Type {
    Type *PointeeTy;
public:
    Type *getPointeeType() const { return PointeeTy; }
    uint64_t getSizeInBits() const override; // 由 DataLayout 决定
};

class StructType : public Type {

    bool Packed;
public:
    StringRef getName() const { return Name; }

};

class FunctionType : public Type {

public:
```

### 2.2 前端适配

#### 2.2.1 C++ 前端：AST → IR 转换层设计

核心类 `ASTToIRConverter`，替代现有 `CodeGenModule` 的 AST→LLVM IR 直接生成：

```cpp
// include/blocktype/Frontend/ASTToIRConverter.h
class ASTToIRConverter {
    btir::BTIRContext &IRCtx;
    btir::Module &IRModule;
    std::unique_ptr<btir::IRBuilder> Builder;

    // 类型映射
    std::unique_ptr<IRTypeMapper> TypeMapper;

    // 值映射：Decl → btir::Value*
    llvm::DenseMap<const Decl*, btir::Value*> DeclValues;
    // 全局变量映射
    llvm::DenseMap<const VarDecl*, btir::GlobalVariable*> GlobalVars;
    // 函数映射
    llvm::DenseMap<const FunctionDecl*, btir::Function*> Functions;
    // 局部变量映射
    llvm::DenseMap<const VarDecl*, btir::AllocaInst*> LocalDecls;

    // 子组件（从现有 CodeGen 模块重构而来）
    std::unique_ptr<IREmitExpr> ExprEmitter;    // 从 CodeGenExpr 重构
    std::unique_ptr<IREmitStmt> StmtEmitter;    // 从 CodeGenStmt 重构
    std::unique_ptr<IREmitCXX> CXXEmitter;      // 从 CGCXX 重构
    std::unique_ptr<IRConstantEmitter> ConstEmitter; // 从 CodeGenConstant 重构

public:
    ASTToIRConverter(btir::BTIRContext &Ctx, btir::Module &M);
    void convert(TranslationUnitDecl *TU);

    // 类型映射接口
    btir::Type *mapType(QualType T);

    // 函数/变量生成
    btir::Function *emitFunction(FunctionDecl *FD);
    btir::GlobalVariable *emitGlobalVar(VarDecl *VD);
};
```

#### 2.2.2 现有 CodeGen 重构为 AST → IR 转换器

**重构映射表**：

| 现有类 | 重构后 | 变化 |
|--------|--------|------|
| `CodeGenModule` | `ASTToIRConverter` | `llvm::Module` → `btir::Module`；`llvm::LLVMContext` → `btir::BTIRContext` |
| `CodeGenFunction` | `IREmitFunction` | `IRBuilderTy` → `btir::IRBuilder`；`llvm::Function*` → `btir::Function*`；`llvm::Value*` → `btir::Value*` |
| `CodeGenTypes` | `IRTypeMapper` | `QualType → llvm::Type*` → `QualType → btir::Type*` |
| `CodeGenConstant` | `IRConstantEmitter` | `llvm::Constant*` → `btir::Constant*` |
| `CodeGenExpr`(99KB) | `IREmitExpr` | 所有 `Emit*Expr` 返回 `btir::Value*` |
| `CodeGenStmt`(50KB) | `IREmitStmt` | 所有 `Emit*Stmt` 使用 `btir::IRBuilder` |
| `CGCXX`(81KB) | `IREmitCXX` | vtable/RTTI/继承使用 `btir::GlobalVariable`/`btir::Function` |
| `CGDebugInfo` | `IRDebugInfo` | `llvm::DIType*` → `btir::Metadata` |
| `Mangler` | `IRMangler` | 保持不变（Mangler 只操作 AST，不依赖 LLVM） |

**关键替换模式**：

```cpp
// 旧：直接生成 LLVM IR
llvm::Value *V = Builder.CreateAdd(LHS, RHS, "addtmp");

// 新：生成 BTIR
btir::Value *V = Builder.createAdd(LHS, RHS, "addtmp");

// 旧：使用 llvm::Type*
llvm::Type *Int32Ty = llvm::Type::getInt32Ty(LLVMCtx);

// 新：使用 btir::Type*
btir::IntegerType *Int32Ty = IRCtx.getInt32Ty();

// 旧：创建 llvm::Function
llvm::Function *F = llvm::Function::Create(FTy, Linkage, Name, TheModule);

// 新：创建 btir::Function
btir::Function *F = IRModule.getOrInsertFunction(Name, FTy);
```

#### 2.2.3 CGCXX（81KB）的拆分和适配策略

CGCXX 是最大的单文件，包含类布局、构造/析构、vtable、RTTI、dynamic_cast、继承、thunk 等逻辑。拆分方案：

```
CGCXX.cpp (81KB) 拆分为：
├── IREmitCXXLayout.cpp    (~15KB)  类布局：ComputeClassLayout, GetFieldOffset, GetClassSize, GetBaseOffset
├── IREmitCXXCtor.cpp      (~15KB)  构造函数：EmitConstructor, EmitBaseInitializer, EmitMemberInitializer
├── IREmitCXXDtor.cpp      (~10KB)  析构函数：EmitDestructor, EmitDestructorBody, EmitDestructorCall
├── IREmitCXXVTable.cpp    (~20KB)  虚函数表：EmitVTable, GetVTableType, GetVTableIndex, InitializeVTablePtr
├── IREmitCXXRTTI.cpp      (~8KB)   RTTI：EmitTypeInfo, EmitCatchTypeInfo
├── IREmitCXXInherit.cpp   (~8KB)   继承：EmitCastToBase, EmitCastToDerived, EmitBaseOffset, EmitDynamicCast
└── IREmitCXXThunk.cpp     (~5KB)   Thunk/VTT：EmitThunk, EmitVTT
```

**适配要点**：
- 所有 `llvm::GlobalVariable*` → `btir::GlobalVariable*`
- 所有 `llvm::Function*` → `btir::Function*`
- 所有 `llvm::Value*` → `btir::Value*`
- 所有 `IRBuilderTy` 操作 → `btir::IRBuilder` 操作
- vtable 布局逻辑不变（它操作的是偏移量计算，与 IR 无关）
- RTTI 结构不变（它生成的是全局常量，只需改类型）

#### 2.2.4 Mangler 的后端无关化

Mangler 当前只依赖 AST 类型和 `CodeGenModule`（用于获取 ASTContext）。重构后：

- **去掉对 `CodeGenModule` 的依赖**：改为依赖 `ASTContext`（Mangler 只需要 AST 信息）
- **Mangler 不需要修改核心逻辑**：它只生成字符串，不涉及 IR 类型
- **签名变更**：`Mangler(CodeGenModule &)` → `Mangler(ASTContext &, const TargetInfo &)`

#### 2.2.5 TargetInfo/ABIInfo 的归属调整

当前 `TargetInfo` 和 `ABIInfo` 在 `CodeGen` 目录下，但它们本质上是**平台信息**而非代码生成逻辑。

**调整方案**：

```
调整前：include/blocktype/CodeGen/TargetInfo.h
调整后：include/blocktype/Target/TargetInfo.h

调整前：include/blocktype/CodeGen/ABIInfo.h
调整后：include/blocktype/Target/ABIInfo.h
```

理由：
- `TargetInfo` 提供类型大小/对齐信息，前端和后端都需要
- `ABIInfo` 提供 ABI 分类，前端（决定参数传递方式）和后端（生成调用代码）都需要
- 移到 `Target` 目录体现其"共享基础设施"地位

## 四、风险评估

### 4.1 编译/测试回归风险

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| IR 指令遗漏导致某些 C++ 特性无法表达 | 中 | 高 | 逐指令对照 LLVM IR 指令集；增量测试 |
| CGCXX 拆分引入 bug | 中 | 高 | 每个子文件迁移后立即测试；保留旧路径对照 |
| 类型映射不一致 | 中 | 中 | IRTypeMapper 和 IRToLLVMConverter 双向测试 |
| 新旧路径并存导致维护负担 | 低 | 中 | 限制并存时间；尽快完成 Phase E |

### 4.2 性能影响评估

| 方面 | 影响 | 分析 |
|------|------|------|
| 编译速度 | 轻微下降（~3-8%） | 引入 IR 中间层增加一次内存拷贝；IR → LLVM IR 转换有开销 |
| 代码质量 | 无影响 | 最终仍使用 LLVM 优化管线，生成代码质量不变 |
| 内存使用 | 增加（~10-20%） | BTIR 和 LLVM IR 同时存在于内存 |
| 二进制大小 | 增加（~5%） | IR 库代码增加 |

**缓解**：Phase E 后可优化 IR → LLVM IR 转换为流式处理，减少内存峰值。

### 4.3 与 Phase 8 工作的兼容性

Phase 8 已完成的工作包括：TargetInfo、ABIInfo、Mangler substitution、Builtins、调用约定、优化 pipeline、目标特定优化。

**兼容性分析**：

| Phase 8 工作 | 兼容方案 |
|--------------|----------|
| TargetInfo/ABIInfo | 移至 `Target/` 目录，前端和后端共享 |
| Mangler substitution | 去掉 CodeGenModule 依赖，改为依赖 ASTContext |
| Builtins | 保持不变，属于前端 AST 层 |
| 调用约定 | 通过 `btir::Function::CallConv` 表达，后端映射为 LLVM CallingConv |
| 优化 pipeline | LLVM 后端内部仍使用 `llvm::PassBuilder`，不受影响 |
| 目标特定优化 | LLVM 后端内部仍使用 `llvm::TargetMachine`，不受影响 |

### 4.4 其他风险

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| IR 设计不够通用，后续前端/后端扩展受限 | 低 | 高 | IR 指令集参考 LLVM IR 和 MLIR，保持通用性 |
| IR 序列化格式不稳定 | 低 | 低 | 初期文本格式为主，二进制格式后续优化 |
| 团队学习成本 | 中 | 中 | 提供详细文档和示例；渐进式引入 |
