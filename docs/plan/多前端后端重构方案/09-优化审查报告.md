### 2.2 自定义 IR vs MLIR 决策验证

**结论：自定义 BTIR 是正确选择。**

| 维度 | 自定义 BTIR | MLIR | 评判 |
|------|-----------|------|------|
| R6 合规 | ✅ IR层不链接LLVM | ❌ MLIR重度依赖LLVM | BTIR胜 |
| 项目规模匹配 | ✅ 5 Dialect足够 | ❌ 30+ Dialect过度 | BTIR胜 |
| C++ 语义精确 | ✅ bt_cpp Dialect精确表达 | ❌ 无现成C++ Dialect | BTIR胜 |
| 渐进式开发 | ✅ 从最小子集开始 | ❌ 需一次性引入框架 | BTIR胜 |
| 编译速度 | ✅ 单层级降级更快 | ❌ 多级Dialect lowering慢 | BTIR胜 |
| 生态复用 | ❌ 需自建基础设施 | ✅ 30+现有Dialect可复用 | MLIR胜 |

### 2.3 BlockType 方案行业定位

**领先点**：TargetLayout独立设计、BackendCapability显式声明、轻量Dialect分层、声明式指令选择规划

**对齐点**：依赖方向单向化、IR层独立库、自有类型/值系统、IR序列化、VerifierPass

**待提升点**：Pass框架缺少分层、Dialect机制过于静态、查询式编译仅远期规划

#### OPT-01：LLVM 依赖分层

**问题**：原方案未区分 LLVM 基础设施依赖（ADT/Support）与 LLVM IR 依赖（IR/Target/Transforms），导致 Sema/AST 层的 27/13 处 ADT 依赖被视为违反 R6。

**修订方案**：引入 `LLVM_ADT_LIBS` 和 `LLVM_IR_LIBS` 分层：

```cmake
llvm_map_components_to_libnames(LLVM_ADT_LIBS support)
llvm_map_components_to_libnames(LLVM_IR_LIBS
  core native asmparser bitwriter irreader transformutils
  x86asmparser x86codegen x86desc x86info
  aarch64asmparser aarch64codegen aarch64desc aarch64info
)
```

重构后依赖关系：

| 库 | 依赖 |
|---|------|
| `libblocktype-basic` | LLVM_ADT_LIBS |
| `libblocktype-ast` | blocktype-basic + LLVM_ADT_LIBS |
| `libblocktype-sema` | blocktype-ast + LLVM_ADT_LIBS |
| `libblocktype-ir` | **blocktype-basic（不链接任何LLVM！）** |
| `libblocktype-frontend` | blocktype-sema + blocktype-ir + LLVM_ADT_LIBS |
| `libblocktype-backend-llvm` | blocktype-ir + LLVM_ADT_LIBS + LLVM_IR_LIBS |

#### OPT-05：Pass 分层设计

**问题**：原方案 `Pass` 基类仅有 `run(IRModule&)` 接口，所有 Pass 都是 Module 级别。

**修订方案**：引入 ModulePass/FunctionPass 分层（参考 LLVM NPM）：

```cpp
class ModulePass {
public:
  virtual ~ModulePass() = default;
  virtual bool runOnModule(IRModule& M) = 0;
  virtual StringRef getName() const = 0;
  virtual InvariantSet getPreservedInvariants() const;
};

class FunctionPass {
public:
  virtual ~FunctionPass() = default;
  virtual bool runOnFunction(IRFunction& F) = 0;
  virtual StringRef getName() const = 0;
  virtual InvariantSet getPreservedInvariants() const;
};

class PassManager {
  bool run(IRModule& M);
  void addPass(std::unique_ptr<ModulePass> P);
  void addPass(std::unique_ptr<FunctionPass> P); // 自动包装为ModulePass
};
```

**优势**：FunctionPass 可自动并行化（不同函数在不同线程优化）。

| 依赖Phase | 工作量 | 验收标准 |
|-----------|--------|---------|
| Phase A（替代原A.6 IRVerifier设计） | +1天 | 1.ModulePass/FunctionPass编译通过 2.PassManager自动包装FunctionPass 3.现有4个Pass迁移到新分层 |

#### OPT-06：VerifierPass 强制化

**问题**：原方案 VerifierPass 是可选的，需要用户手动添加。

**修订方案**：参考 Swift SILVerifier，Debug 构建自动验证：

```cpp
bool PassManager::run(IRModule& M) {
  for (auto& Entry : Passes) {
    bool Changed = Entry.MP ? Entry.MP->runOnModule(M)
                             : /* FunctionPass wrapper */;
    if (DebugMode) {
      VerifierPass V;
      auto Preserved = Entry.MP ? Entry.MP->getPreservedInvariants()
                                 : Entry.FP->getPreservedInvariants();
      if (!V.verify(M, Preserved)) {
        Diags.report(IRPassInvariantBroken, Entry.MP->getName());
      }
    }
  }
}
```

增加 `InvariantSet` 机制：每个 Pass 声明保持的不变量，VerifierPass 仅检查被声明为"不保持"的不变量，减少验证开销。

| 依赖Phase | 工作量 | 验收标准 |
|-----------|--------|---------|
| Phase A（与OPT-05一起） | +0.5天 | 1.Debug构建自动在每个Pass后运行VerifierPass 2.InvariantSet机制工作正常 3.Release构建无额外开销 |

#### OPT-07：IRVisitor 模式

**问题**：原方案 IRConsumer 仅有 `processModule/processFunction/processInstruction` 三个方法，后端需自行按指令类型分派。

**修订方案**：新增 IRVisitor 高级接口（参考 LLVM InstVisitor），IRConsumer 保留为简单接口：

```cpp
class IRVisitor {
public:
  virtual ~IRVisitor() = default;
  virtual void visitModule(IRModule& M);
  virtual void visitFunction(IRFunction& F);
  virtual void visitBasicBlock(IRBasicBlock& BB);
  virtual void visitInstruction(IRInstruction& I); // fallback

  // 按指令类型自动分派
  virtual void visitRet(IRInstruction& I);
  virtual void visitBr(IRInstruction& I);
  virtual void visitCondBr(IRInstruction& I);
  virtual void visitAlloca(IRInstruction& I);
  virtual void visitLoad(IRInstruction& I);
  virtual void visitStore(IRInstruction& I);
  virtual void visitAdd(IRInstruction& I);
  virtual void visitCall(IRInstruction& I);
  virtual void visitICmp(IRInstruction& I);
  virtual void visitPhi(IRInstruction& I);
  // ... 每种指令类型一个visit方法

  void visit(IRModule& M);   // 遍历入口
  void visit(IRFunction& F);
  void visit(IRBasicBlock& BB);
};
```

IRToLLVMConverter 继承 IRVisitor，按指令类型实现对应的 LLVM IR 生成，替代原 `convertInstruction()` 中的大型 switch-case。

| 依赖Phase | 工作量 | 验收标准 |
|-----------|--------|---------|
| Phase C（IRToLLVMConverter设计时） | +1天 | 1.IRVisitor编译通过 2.IRToLLVMConverter继承IRVisitor 3.无switch-case指令分派 |

#### OPT-08：TargetLayout 字段明确化

**问题**：原方案 TargetLayout 为抽象接口，未明确字段和数据来源。

**修订方案**：明确 TargetLayout 为 `llvm::DataLayout` 的子集，仅含 IR 层需要的布局信息：

```cpp
class TargetLayout {
public:
  uint32_t PointerSize;
  uint32_t PointerAlign;
  uint32_t CharWidth;
  uint32_t IntAlign;
  uint32_t LongAlign;
  uint32_t LongLongAlign;
  uint32_t FloatAlign;
  uint32_t DoubleAlign;
  uint32_t LongDoubleAlign;
  uint32_t SizeType;
  uint32_t PtrDiffType;
  bool IsBigEndian;

  static TargetLayout fromTargetInfo(const TargetInfo& TI);

  uint64_t getTypeSizeInBits(const IRType* T) const;
  uint64_t getTypeAlignInBits(const IRType* T) const;
  uint64_t getTypeAllocSizeInBits(const IRType* T) const;
};
```

构造路径：`TargetTriple → TargetInfo → TargetLayout`（TargetLayout 不持有 `llvm::DataLayout` 引用）。

| 依赖Phase | 工作量 | 验收标准 |
|-----------|--------|---------|
| Phase A（替代原A.2 TargetLayout设计） | 0天（原设计细化） | 1.TargetLayout字段完整 2.fromTargetInfo()正确映射 3.不持有llvm::DataLayout引用 |

#### OPT-09：DialectRegistry 动态注册

**问题**：原方案 5 个固定 Dialect 通过 `DialectID` 枚举定义，新增 Dialect 需修改核心头文件。

**修订方案**：保留 5 个内置 Dialect，增加 `DialectRegistry` 动态注册：

```cpp
class DialectRegistry {
  DenseMap<StringRef, DialectFactory> Registry;
  DenseMap<StringRef, uint16_t> NameToID;
  uint16_t NextUserID = 256;
public:
  static DialectRegistry& instance();
  void registerDialect(StringRef Name, DialectFactory F);
  Dialect* getDialect(StringRef Name);
  uint16_t getDialectID(StringRef Name);

  static constexpr uint16_t BtCoreID = 0;
  static constexpr uint16_t BtCppID = 1;
  static constexpr uint16_t BtTargetID = 2;
  static constexpr uint16_t BtDebugID = 3;
  static constexpr uint16_t BtMetaID = 4;
  static constexpr uint16_t FirstUserID = 256;
};
```

未来可扩展 Dialect：`bt_coroutine`（C++20协程）、`bt_concept`（C++20概念）、`bt_reflection`（C++26反射）。

#### OPT-10：Frontend/Backend 注册生命周期管理

**问题**：原方案 FrontendRegistry/BackendRegistry 支持注册和查找，但未说明注册时机、销毁时机和线程安全。

## 五、风险评估修订

### 5.1 CGCXX 拆分风险降级

**原评估**：CGCXX 拆分是关键瓶颈，风险中，5天工期。

**修订**：实际 CGCXX.cpp 仅 2,229 行，拆分风险从"中"降级为"低"，工期从 5 天缩减为 2-3 天。**CGCXX 拆分不再是关键瓶颈。**

### 5.2 新增风险：CompilerInstance 重构

| 风险 | 概率 | 影响 | 缓解措施 | 检测方法 |
|------|------|------|---------|---------|
| CompilerInstance LLVM依赖移除不完整 | 中 | 高 | 分类处理（OPT-03）：管线编排保留、LLVM上下文/优化/代码生成移入LLVMBackend | CompilerInstance.h不再包含任何`llvm/IR/*.h`头文件 |
| 新旧管线切换导致编译行为差异 | 中 | 中 | 差分测试管线（OPT-04）自动对比新旧路径编译结果 | CI差分测试自动运行 |

### 5.3 新增风险：LLVM 依赖分层

| 风险 | 概率 | 影响 | 缓解措施 | 检测方法 |
|------|------|------|---------|---------|
| CMake依赖分层导致链接错误 | 低 | 中 | 渐进式分层：先分离IR依赖，再分离ADT依赖 | 每个库独立编译链接成功 |
| Sema/AST的ADT依赖被误判为IR依赖 | 低 | 低 | 代码审查：逐一确认每个LLVM include的类别 | `grep -c 'llvm/IR/' include/blocktype/Sema/` = 0 |

### 5.4 关键路径修订

**原关键路径**：Phase A → Phase B (CGCXX 拆分) → Phase D → Phase E

**修订后关键路径**：Phase A → Phase B (前端适配) → Phase D (CompilerInstance 重构) → Phase E

CompilerInstance 重构（OPT-03）成为新的关键瓶颈。
