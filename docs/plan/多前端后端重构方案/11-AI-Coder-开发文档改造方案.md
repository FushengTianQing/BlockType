# AI Coder 开发文档改造方案

## 问题诊断

### 原始文档结构

```
00-对比分析与融合决策.md    ← 决策依据（AI coder 不需要）
01-总体架构设计.md          ← 接口定义（AI coder 需要但看不到）
02-IR层实现.md             ← 接口定义（AI coder 需要但看不到）
03-前端适配.md             ← 接口定义（AI coder 需要但看不到）
04-后端适配.md             ← 接口定义（AI coder 需要但看不到）
05-渐进式改造方案-PhaseAB.md ← 执行计划（但缺少接口定义）
06-渐进式改造方案-PhaseCDE.md ← 执行计划（但缺少接口定义）
07-风险评估与工作量估算.md   ← AI coder 不需要
08-附录.md                 ← AI coder 不需要
09-优化审查报告.md          ← 优化任务（但缺少接口定义）
10-渐进式改造方案-PhaseFGH.md ← 执行计划（但缺少接口定义）
```

### 核心问题

**执行计划（05/06/10/09）只说了"做什么"，但 AI coder 执行时需要的"怎么做"的具体约束——接口签名、类型定义、参数规格——全在 01-04 文档里。AI coder 执行 Task 时看不到这些定义，只能自己编造。**

具体表现：

| 缺失项 | 后果 | 示例 |
|--------|------|------|
| 无接口签名 | AI coder 自行设计类接口，与架构设计不一致 | Task A.1 说"实现 IRType 体系"，但 IRType 的 Kind 枚举值、虚函数列表在 02 文档里 |
| 无参数约束 | AI coder 自行决定参数类型和名称 | Task B.3 说"实现 IRTypeMapper"，但 mapType 的映射规则表在 03 文档里 |
| 无枚举精确值 | AI coder 自行分配枚举值，与序列化格式不兼容 | Opcode 枚举的分组和数值在 02 文档里 |
| 无类型映射规则 | AI coder 自行实现映射逻辑 | IRType→llvm::Type 的映射规则在 04 文档里 |
| 无语义约束 | AI coder 不知道 getOrInsertFunction 的精确行为 | "按名查找，类型匹配返回已有，类型不匹配返回 nullptr"在 02 文档里 |

### 原始 Task 格式（9 列表）

```
| Task | 内容 | 预估 | 输入 | 输出 | 实现约束 | 验收标准 | 测试用例 | 依赖文件 |
```

问题：
1. "输入"列写的是概念名（如"blocktype-basic ADT"），不是具体文件路径
2. "实现约束"是自然语言描述，不是可检查的断言
3. "验收标准"是自然语言，不是可执行命令
4. 缺少接口签名——AI coder 不知道要实现什么类、什么方法
5. 附加增强任务只有 4 列，信息更少

---

## 改造方案

### 核心原则：自包含

**每个 Task 必须自包含所有 AI coder 编码所需的信息，不需要跨文档查找。**

具体做法：把 01-04 文档中的接口定义、枚举值、映射规则表等，**内联到每个 Task 中**。

### 新 Task 格式

每个 Task 包含以下固定结构：

```markdown
## Task X.Y：任务标题

### 依赖
列出必须先完成的 Task 编号

### 产出文件
| 操作 | 文件路径 |
新增/修改/删除的具体文件

### 必须实现的类型定义
完整的 class/struct/enum/函数签名（从 01-04 文档中提取，内联到此）

### 实现约束
可检查的硬约束列表

### 验收标准
可执行的断言代码（不是自然语言描述）
```

此外，在每个 Phase 任务流文档的**开头**（执行规则之后、第一个 Task 之前），必须包含：

```markdown
## 接口关联关系

### 持有关系（A 持有 B 的指针/引用/unique_ptr）
列出本 Phase 所有接口之间的持有/引用关系，标注方向和语义

### 调用关系（A 的方法调用 B 的方法）
列出本 Phase 所有方法之间的调用关系，标注精确语义

### 生命周期约束
列出关键对象的生命周期大小关系

### 内存管理规则
列出每个对象的分配方式、拥有者、释放方式
```

**为什么需要关联关系？** 没有它，AI coder 实现出来的就是一堆孤立的类——IRPointerType 不知道 PointeeType 指向 IRTypeContext 管理的类型，IRBuilder 不知道 createAdd 要调用 IRContext::create 再调用 InsertBB->push_back，IRModule 不知道 getOrInsertFunction 的精确语义。关联关系把这些接口连成一张网，AI coder 才能实现出互相连通的功能。

### 改造对照表

| 原始格式 | 新格式 | 改造内容 |
|---------|--------|---------|
| "输入"列：概念名 | "依赖"：Task 编号 | 从概念依赖改为 Task 依赖 |
| "输出"列：文件名 | "产出文件"：操作+路径 | 明确新增/修改/删除 |
| "实现约束"：自然语言 | "必须实现的类型定义"：完整代码签名 | 从描述改为精确代码 |
| 无 | "实现约束"：可检查断言 | 新增 |
| "验收标准"：自然语言 | "验收标准"：可执行断言代码 | 从描述改为代码 |
| "测试用例"列 | 合并到"验收标准" | 去重 |
| 无 | 枚举精确值、映射规则表 | 从 01-04 内联 |
| 无 | 语义约束（如 getOrInsertFunction 行为） | 从 02 内联 |

### 文件命名规范

```
11-AI-Coder-任务流-PhaseA.md    ← Phase A 的 AI coder 任务流
12-AI-Coder-任务流-PhaseB.md    ← Phase B
13-AI-Coder-任务流-PhaseCDE.md  ← Phase C/D/E
14-AI-Coder-任务流-PhaseFGH.md  ← Phase F/G/H
15-AI-Coder-任务流-优化Phase.md  ← 09 的 OPT 任务
```

### 原始文档保留

01-04 文档保留作为**设计参考**，但 AI coder 执行时**不需要查阅**。所有必要信息已内联到任务流文档中。

05/06/10/09 文档保留作为**人类阅读的概览**，但 AI coder 执行时**使用 11-15 文档**。

## 执行规则

AI coder 执行任务流时必须遵守：

1. **严格按 Task 编号顺序执行**，每个 Task 完成并通过验收后再开始下一个
2. **接口签名不可修改**——任务流文档中的 class/struct/enum/函数签名是硬约束
3. **验收标准必须全部通过**——验收代码是可执行的断言
4. **命名空间**：IR 层代码在 `namespace blocktype::ir`，前端层在 `namespace blocktype::frontend`，后端层在 `namespace blocktype::backend`
5. **头文件路径**：`include/blocktype/IR/`、`include/blocktype/Frontend/`、`include/blocktype/Backend/`
6. **依赖限制**：`libblocktype-ir` 仅依赖 `blocktype-basic`，不链接 LLVM
