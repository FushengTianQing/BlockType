### 1.1.2 各层职责边界

| 层次 | 职责 | 输入 | 输出 | 核心接口 |
|------|------|------|------|----------|
| **前端层** | 源代码 → AST → BTIR | 源代码+LangOpts | `unique_ptr<ir::IRModule>` | `FrontendBase::compile()` |
| **IR 中间层** | 类型/值/指令的统一表示 | 前端产出的IR | 后端可消费的IR | `IRModule`, `IRBuilder`, `IRPass` |
| **后端层** | BTIR → 目标代码 | `const IRModule&`+TargetOpts | 目标文件/汇编 | `BackendBase::emitObject()` |

### 1.1.3 解耦策略

1. **依赖方向单向化**：`Frontend → BTIR ← Backend`，前端和后端之间零依赖
2. **接口抽象**：前端通过 `ir::IRBuilder` 构建 IR；后端通过 `IRConsumer` 遍历 `ir::IRModule` 消费 IR
3. **类型系统独立**：BTIR 拥有自有的类型系统（`ir::IRType`），与 `llvm::Type` 和 `QualType` 均独立
4. **值系统独立**：BTIR 拥有自有的值系统（`ir::IRValue`+`Use`/`User`），与 `llvm::Value` 独立
5. **布局信息独立**：`TargetLayout` 不依赖 LLVM `DataLayout`，IR层可独立计算类型大小
6. **IR层作为独立库**：`libblocktype-ir` 不链接 LLVM，仅依赖 `blocktype-basic`

### 1.1.4 自由组合机制

```bash
bt --frontend=cpp --backend=llvm -O2 source.cpp -o source.o
bt --frontend=cpp --backend=cranelift source.cpp -o source.o
bt --frontend=rust --backend=llvm source.rs -o source.o
bt --frontend=cpp --emit-btir source.cpp -o source.btir
bt --backend=llvm source.btir -o source.o
bt --frontend=cpp --verify-btir source.cpp
```

```cpp
auto Frontend = FrontendRegistry::instance().create("cpp", Opts, Diags);
auto Backend  = BackendRegistry::instance().create("llvm", Opts, Diags);
auto IRModule = Frontend->compile(Source, TypeCtx, Layout);
Backend->emitObject(*IRModule, OutputPath);
```

## 1.2 中间表示（IR）层设计

### 1.2.1 核心决策：自定义 BTIR

| 方案 | 优势 | 劣势 | 评分 |
|------|------|------|------|
| **自定义IR(BTIR)** | 完全可控、轻量、无外部依赖、精确匹配语义 | 需自建类型系统/验证器 | ★★★★★ |
| MLIR | 成熟框架、多级Dialect | 重度依赖LLVM、与解耦目标矛盾 | ★★ |
| LLVM IR | 无需额外实现 | 与LLVM硬耦合，违反R6 | ★ |
| Cranelift IR | 轻量 | Rust生态绑定、C++语义支持弱 | ★★ |

**选择：自定义IR（BTIR）**。理由：R6合规、R1合规、轻量渐进、已有先例（GCC GIMPLE/Rust MIR/Swift SIL）。

### 1.2.2 IR 类型系统

BTIR采用自定义类型系统，独立于`llvm::Type`和`QualType`。

- 核心类型：`IRVoidType`/`IRIntegerType`/`IRFloatType`/`IRPointerType`/`IRArrayType`/`IRStructType`/`IRFunctionType`/`IRVectorType`/`IROpaqueType`
- 类型唯一化：`IRTypeContext`（`DenseMap`/`FoldingSet`实现），拥有所有`IRType`实例生命周期
- 类型映射：`IRTypeMapper::mapType(QualType) → ir::IRType*`（前端）；`LLVMTypeMapper::mapType(ir::IRType*) → llvm::Type*`（后端）

完整实现规范 → 见02-IR层实现.md §2.1.3-2.1.4

### 1.2.3 IR 指令集

SSA 形式，后端无关：

| 类别 | 操作码 |
|------|--------|
| 终结 | Ret, Br, CondBr, Switch, Unreachable, Invoke |
| 整数二元 | Add, Sub, Mul, UDiv, SDiv, URem, SRem |
| 浮点二元 | FAdd, FSub, FMul, FDiv, FRem |
| 位运算 | Shl, LShr, AShr, And, Or, Xor |
| 内存 | Alloca, Load, Store, GEP, Memcpy |
| 转换 | Trunc, ZExt, SExt, FPTrunc, FPExt, FPToSI, SIToFP, PtrToInt, IntToPtr, BitCast |
| 比较 | ICmp, FCmp |
| 调用 | Call, Invoke |
| 其他 | Phi, Select, ExtractValue, InsertValue |
| 调试 | DbgDeclare, DbgValue |

C++ 特有语义（vtable/RTTI/dynamic_cast）通过**函数调用+约定**表达，而非特殊指令。

### 1.2.4 IR 值系统

BTIR值系统采用SSA形式，独立于`llvm::Value`。

- 值层次：`IRValue` → `IRInstruction`/`IRArgument`/`IRConstant`*/`IRBasicBlock`
- Use-Def/Def-Use链：`ir::Use{IRValue*, User*}` + `ir::User: public IRValue{vector<Use>}`
- 常量类：`IRConstantInt`/`IRConstantFP`/`IRConstantNull`/`IRConstantStruct`/`IRConstantArray`/`IRConstantUndef`/`IRConstantZero`/`IRConstantAggregateZero`/`IRConstantFunctionRef`/`IRConstantGlobalRef`

完整实现规范 → 见02-IR层实现.md §2.1.5

### 1.2.5 IR 模块层次

- `IRModule(Name, TargetTriple, DataLayoutStr)` 拥有 `IRFunction[]`/`IRFunctionDecl[]`/`IRGlobalVariable[]`/`IRGlobalAlias[]`/`IRMetadata[]`
- `IRFunction(Name, Linkage, CallingConv, Attributes, Arguments[], BasicBlocks[])`
- `IRBasicBlock` → `IRInstruction[]`（SSA指令链表）

完整实现规范 → 见02-IR层实现.md §2.1.6

### 1.2.7 IR 构建器

`IRBuilder`集成常量工厂和所有指令创建方法，核心接口包括：
- 常量工厂：`getInt1/getInt32/getInt64/getNull`
- 终结指令：`createRet/createBr/createCondBr/createInvoke`
- 算术指令：`createAdd/createSub/createMul/createNeg`
- 比较指令：`createICmp/createFCmp`
- 内存指令：`createAlloca/createLoad/createStore/createGEP`
- 转换指令：`createBitCast/createZExt/createSExt/createTrunc`
- 调用指令：`createCall`
- 聚合操作：`createExtractValue/createInsertValue`

完整实现规范 → 见02-IR层实现.md §2.1.7

### 1.2.8 IR 验证器

`VerifierPass`检查IR正确性：类型完整性、SSA性质、终结指令、类型匹配、调用参数、引用完整性。

完整实现规范 → 见02-IR层实现.md §2.1.8

### 1.2.9 IR 序列化

`IRWriter`/`IRReader`支持文本格式（人类可读）和二进制格式（紧凑高效）。

完整实现规范 → 见02-IR层实现.md §2.1.9

文本格式示例：
```
; BTIR text format
module "test" target "x86_64-unknown-linux-gnu"

global @g1 : i32 = 42

function @add(i32 %a, i32 %b) -> i32 {
entry:
  %1 = add i32 %a, %b
  ret i32 %1
}

function @main() -> i32 {
entry:
  %1 = call @add(i32 1, i32 2)
  ret i32 %1
}
```

### 1.2.13 IR 内存管理策略

**设计决策**：BumpPtrAllocator + unique_ptr 混合策略

- IR内部节点（IRInstruction/IRValue/IRType）用BumpPtrAllocator分配（O(1)分配，批量释放）
- IRModule作为顶层对象用unique_ptr拥有
- `IRContext`统一管理分配器和类型唯一化上下文

内存管理职责划分：

| 对象 | 分配方式 | 生命周期 | 拥有者 |
|------|---------|---------|--------|
| `IRContext` | 栈/unique_ptr | 编译过程 | `CompilerInstance` |
| `IRType` 及子类 | `IRContext::create<T>()` | 随IRContext销毁 | `IRTypeContext` |
| `IRValue`/`IRInstruction` | `IRContext::create<T>()` | 随IRContext销毁 | `IRBasicBlock` |
| `IRModule` | `unique_ptr<IRModule>` | 编译过程 | `CompilerInstance` |
| `IRFunction`/`IRBasicBlock` | `unique_ptr` | 随IRModule销毁 | `IRModule` |

完整实现规范 → 见02-IR层实现.md §2.1.11

### 1.2.14 IR 线程安全设计

**设计决策**：初期单线程 + 预留接口

- `IRThreadingMode`枚举：`SingleThread`（默认）/ `MultiInstance` / `SharedReadOnly`
- Phase A-E使用`SingleThread`，未来扩展为多实例和共享只读模式

完整实现规范 → 见02-IR层实现.md §2.1.12

### 1.2.15 IR 错误恢复机制

**设计决策**：复用DiagnosticsEngine + IR专用错误结果类型

- `IRConversionResult`：参考Sema的ExprResult/StmtResult模式
- `IRVerificationResult`：收集所有违规一次性报告
- 错误恢复策略：类型映射失败用`IROpaqueType`占位、语句转换失败跳过、表达式转换失败用`IRConstantUndef`占位

完整实现规范 → 见02-IR层实现.md §2.1.13

### 1.2.16 IR 版本兼容性

**设计决策**：语义版本 + 二进制格式版本号

- `IRFormatVersion{Major, Minor, Patch}`：主版本不兼容、次版本向后兼容、补丁完全兼容
- `IRFileHeader`：魔数"BTIR" + 版本号 + 标志位 + 偏移量

完整实现规范 → 见02-IR层实现.md §2.1.14

### 1.2.17 后端能力声明

**设计决策**：枚举特性集 + 运行时查询接口

- `IRFeature`位掩码枚举：IntegerArithmetic/FloatArithmetic/VectorOperations/AtomicOperations/ExceptionHandling/DebugInfo/VarArg/SeparateFloatInt/StructReturn/DynamicCast/VirtualDispatch/Coroutines
- `BackendCapability`：`declareFeature/hasFeature/supportsAll/getUnsupported`

完整实现规范 → 见02-IR层实现.md §2.1.15

### 1.2.18 增量编译支持（远期预留接口）

`IncrementalCompilationState`远期预留接口，支持依赖追踪和IR缓存。

渐进式路线：Phase A-E无增量编译 → 未来1: IR缓存 → 未来2: 自动增量 → 未来3: IR差异合并

## 1.5 编译管线编排

### 1.5.1 CompilerInstance 重构

```cpp
class CompilerInstance {
  shared_ptr<CompilerInvocation> Invocation;
  unique_ptr<DiagnosticsEngine> Diags;
  unique_ptr<FrontendBase> Frontend;
  unique_ptr<BackendBase> Backend;
  unique_ptr<ir::IRContext> IRCtx;          /// IR 内存管理上下文（新增）
  unique_ptr<ir::IRModule> IRModule;
  unique_ptr<ir::IRTypeContext> IRTypeCtx;
  unique_ptr<TargetLayout> Layout;
public:
  bool compileFile(StringRef Filename);
private:
  bool runFrontend(StringRef Filename);  // 源代码 → BTIR
  bool runIRPipeline();                   // BTIR 验证/优化
  bool runBackend(StringRef OutputPath);  // BTIR → 目标代码
  bool runLinker(const vector<string>& ObjFiles, StringRef OutputPath);
};
```

### 1.5.2 管线流程

```
compileFile("source.cpp"):
  1. 解析选项 → createFrontend("cpp") + createBackend("llvm")
  2. runFrontend → Preprocessor → Parser → Sema → AST → ASTToIRConverter → ir::IRConversionResult
     - 如果 IRConversionResult.isInvalid() → 报告错误，中止或继续（取决于错误恢复策略）
  3. 契约验证 → contract::verifyAllContracts(IRModule)（Debug 构建默认启用）
  4. runIRPipeline → VerifierPass + OptimizationPasses
  5. 后端能力检查 → Backend->getCapability().supportsAll(IRModule.getRequiredFeatures())
     - 不支持的特性 → 报告错误，中止或降级
  6. runBackend → IRToLLVMConverter → llvm::Module → LLVM优化 → 目标文件
  7. runLinker → 链接可执行文件
```

### 1.5.3 新增编译选项

```
--frontend=<name>     选择前端，默认自动选择
--backend=<name>      选择后端，默认 llvm
--emit-btir           输出 BTIR 文本格式
--emit-btir-bitcode   输出 BTIR 二进制格式
--btir-only           只输出 BTIR，不运行后端
--load-btir=<file>    从 BTIR 文件加载，跳过前端
--verify-btir         验证 IR 正确性
```

## 1.6 行业前沿增强架构总览

### 1.6.2 增强子系统与现有架构的关系

| 增强子系统 | 所属层 | 与现有架构的关系 | 引入阶段 |
|-----------|--------|-----------------|---------|
| **Dialect 分层机制** | IR层 | 为IRInstruction/IRType添加DialectID字段，扩展BackendCapability | Phase A-B |
| **Telemetry 遥测系统** | 编译管线 | 新增TelemetryCollector，通过PhaseGuard RAII与CompilerInstance集成 | Phase A-B |
| **Plugin 插件系统** | 编译管线 | 上层管理器，FrontendRegistry/BackendRegistry保持不变 | Phase C-F |
| **Cache 缓存架构** | 编译管线 | 新增CompilationCacheManager，与IR序列化集成 | Phase B-E |
| **Security 安全架构** | IR层 | 扩展IRModule（可重现模式）+ 新增IRIntegrityChecksum | Phase A-E |
| **FFI 跨语言互操作** | IR层 | 新增FFIFunctionDecl/FFITypeMapper，扩展IR指令集 | Phase C-E |
| **Diag 诊断升级** | 编译管线 | 扩展DiagnosticsEngine，新增StructuredDiagEmitter | Phase B-C |
| **AI 辅助预留** | IR层 | 新增IRStatistics + AIAutoTuner接口（远期） | Phase H |
| **Testing 测试升级** | 测试框架 | 新增DifferentialTester/IRFuzzer/IREquivalenceChecker | Phase D-E |
| **DebugInfo 调试信息升级** | IR层 | 扩展IRInstruction（DbgInfo字段）+ 新增IRDebugMetadata | Phase A-D |
| **Query 查询式编译** | 编译管线 | 新增QueryContext，为增量编译奠基（远期） | Phase F-G |

**设计原则**：
- 所有增强子系统**不改变现有接口**，仅通过扩展字段或新增接口实现
- 所有增强子系统**保持前端/后端独立替换**（R2合规）
- 所有增强子系统**分阶段引入**，每阶段可编译可测试（R3合规）
- 所有增强子系统**通过抽象接口交互**（R5合规）

## 1.7 IR层增强设计概要

> 从09-行业前沿补充优化方案提取各增强子系统的概要设计，仅保留接口定义和关键设计决策。完整代码实现见02-IR层实现等具体技术方案文档。

### 1.7.1 Dialect 分层机制

**设计动机**：当前BTIR是单一层级IR，C++特有语义与通用算术指令混合，后端无法选择性消费特定语义层。

**关键设计决策**：引入轻量Dialect分层（借鉴MLIR但更简洁），5个固定Dialect覆盖所有需求。

**核心接口**：`DialectID`枚举（Core/Cpp/Target/Debug/Metadata）、`IRInstruction`/`IRType`扩展Dialect字段、`DialectCapability`能力声明、`DialectLoweringPass`降级管线

**Dialect分层映射**：bt_core（通用算术/控制流/内存）→ bt_cpp（C++语义）→ bt_target（目标特定）→ bt_debug（调试信息）→ bt_meta（编译器元数据）

**Dialect Lowering管线**：前端产出bt_core+bt_cpp+bt_debug+bt_meta → DialectLoweringPass(bt_cpp→bt_core) → 后端消费

完整实现规范 → 见02-IR层实现.md §2.1.19

### 1.7.2 查询式编译架构

**设计动机**：当前增量编译仅为远期预留接口（`IncrementalCompilationState`），缺乏具体架构设计。

**关键设计决策**：引入查询式编译架构（参考Rustc），为增量编译奠定基础。

**核心接口**：`QueryKey`/`QueryResult<T>`/`QueryContext`/编译查询定义/投影查询模式/红绿标记算法

渐进式路线：Phase A-E无查询系统 → Phase F基础查询 → Phase G增量编译 → Phase H自动增量

完整实现规范 → 见02-IR层实现.md §2.1.20

### 1.7.3 编译器可观测性

**设计动机**：当前编译器无任何性能遥测或自剖析能力，编译慢时无法定位瓶颈。

**关键设计决策**：引入编译器自剖析框架（参考LLVM `-ftime-report` 和 Rustc `-Zself-profile`），通过RAII自动记录编译阶段耗时。

**核心接口**：`CompilationPhase`枚举、`CompilationEvent`记录、`TelemetryCollector`收集器、`PhaseGuard` RAII守卫

与CompilerInstance集成：新增`Telemetry`成员，编译各阶段通过`scopePhase`自动计时

完整实现规范 → 见02-IR层实现.md §2.1.21

### 1.7.4 编译器安全架构

**设计动机**：当前方案无安全相关设计，编译器作为关键基础设施需要IR完整性保证、编译过程可重现性、IR Pass正确性验证。

**关键设计决策**：引入三层安全机制——IR完整性验证、可重现构建、Pass不变量检查。

**核心接口**：`IRIntegrityChecksum`校验和、`IRSigner`签名、`PassInvariantChecker`不变量检查、`IRModule`可重现模式

完整实现规范 → 见02-IR层实现.md §2.1.22

### 1.7.5 编译器插件系统

**设计动机**：当前方案有FrontendRegistry/BackendRegistry但无插件系统，无法动态加载第三方扩展。

**关键设计决策**：引入三层插件架构（IR Pass插件 + 前端插件 + 后端插件），参考LLVM New Pass Manager + Clang Plugin。FrontendRegistry/BackendRegistry保持不变，插件系统是上层管理器。

**核心接口**：`PluginType`枚举、`PluginInfo`元数据、`CompilerPlugin`基类、`PluginManager`管理器

渐进式路线：Phase A-E仅预留接口 → Phase F基础插件加载 → Phase G IR Pass插件注册

### 1.7.6 编译器诊断升级

**设计动机**：当前方案使用`DiagnosticsEngine`但缺乏结构化诊断输出（JSON/SARIF）、FixIt Hint、诊断交叉引用和分组管理。

**关键设计决策**：升级诊断系统为行业领先水平，扩展DiagnosticsEngine而非替换。

**核心接口**：`StructuredDiagnostic`（含FixIt/Notes/IR扩展）、`StructuredDiagEmitter`（SARIF/JSON/Text）、`DiagnosticGroupManager`

### 1.7.7 编译器缓存架构

**设计动机**：当前方案增量编译仅有远期接口，缺乏编译缓存架构设计。

**关键设计决策**：引入多层编译缓存（参考sccache/ccache），支持本地磁盘和远程缓存（远期）。

**核心接口**：`CacheKey`哈希摘要、`CacheEntry`缓存条目、`CacheStorage`可插拔存储接口、`LocalDiskCache`/`RemoteCache`、`CompilationCacheManager`管理器

完整实现规范 → 见02-IR层实现.md §2.1.23

### 1.7.8 AI辅助编译预留

**设计动机**：Compiler-R1(2025)已证明RL驱动的Pass序列自动调优可行，需预留接口。

**关键设计决策**：远期Phase H实现，当前仅定义接口。IR层预留AutoPhase式统计特征接口。

**核心接口**：`IRStatistics`统计特征、`PassSequenceEvaluation`评估结果、`AIAutoTuner`调优接口

完整实现规范 → 见02-IR层实现.md §2.1.26

### 1.7.9 编译器测试升级

**设计动机**：当前测试以单元测试和集成测试为主，缺乏编译器专用测试方法论（差分测试、模糊测试、等价性验证）。

**关键设计决策**：引入编译器行业级测试三件套。

**核心接口**：`DifferentialTester`差分测试、`IRFuzzer`模糊测试、`IREquivalenceChecker`等价性检查

### 1.7.10 跨语言互操作

**设计动机**：当前方案仅支持C++前端，缺乏与其他语言互操作的设计。

**关键设计决策**：在IR层设计FFI（Foreign Function Interface）接口，优先支持C ABI（零开销），其他语言通过C ABI桥接。

**核心接口**：`CallingConvention`枚举、`FFITypeMapper`类型映射、`FFIFunctionDecl`函数声明、`FFIModule`模块管理、`FFIOpcode`指令扩展

完整实现规范 → 见02-IR层实现.md §2.1.24

### 1.7.11 调试信息升级

**设计动机**：当前方案调试信息生成完全委托LLVM，缺乏IR层调试元数据设计。

**关键设计决策**：在IR层建立调试元数据体系，支持DWARF5标准。扩展IRInstruction添加DbgInfo字段。

**核心接口**：`IRDebugMetadata`（SourceLocation/DIType/DISubprogram/DICompileUnit）、`IRInstructionDebugInfo`指令调试标注、`DebugInfoEmitter`发射器接口

完整实现规范 → 见02-IR层实现.md §2.1.25

## 1.8 行业标准对标

> 将BlockType编译器与行业主流编译器进行系统性对标，明确各特性的对标程度和关键设计决策。

### 1.8.1 LLVM/Clang 对标

| 特性 | LLVM/Clang 实现 | BlockType 对标方案 | 对标程度 |
|------|----------------|-------------------|---------|
| 多级IR | LLVM IR (单一层级) + MIR (机器IR) | BTIR Dialect分层(bt_core/bt_cpp/bt_target) | **超越**：Dialect机制比LLVM单一IR更灵活 |
| Pass管线 | New Pass Manager (模块化Pass管线) | IR PassManager + DialectLoweringPass | **对齐**：Pass管线设计对齐NPM |
| 增量编译 | ThinLTO + 增量模块编译 | 查询式编译 + 红绿标记 | **超越**：查询式架构比LLVM增量编译更精细 |
| 诊断系统 | Clang Diagnostics (FixIt + 模板差异) | StructuredDiagnostic + FixIt + SARIF | **对齐+增强**：增加SARIF格式 |
| 调试信息 | LLVM DebugInfo Metadata (DWARF5) | IRDebugMetadata + DWARF5 + CodeView | **对齐** |
| 插件系统 | New Pass Manager Plugin | CompilerPlugin + PluginManager | **对齐** |
| 缓存系统 | 无内置缓存 | CompilationCacheManager (本地+远程) | **超越**：LLVM无内置编译缓存 |
| 遥测 | -ftime-report + -ftime-trace | TelemetryCollector + Chrome Trace | **对齐** |
| 模糊测试 | Bugpoint + lit测试 | IRFuzzer + DifferentialTester + IREquivalenceChecker | **超越**：三件套覆盖更全面 |
| AI调优 | MLGO (ML引导内联/寄存器分配) | AIAutoTuner (RL驱动Pass序列) | **超越**：Pass序列级调优比单Pass调优更全面 |

### 1.8.2 MLIR 对标

| 特性 | MLIR 实现 | BlockType 对标方案 | 对标程度 |
|------|----------|-------------------|---------|
| Dialect机制 | 完整Dialect基础设施(定义/验证/变换/降级) | 轻量DialectID + DialectLoweringPass | **部分对齐**：BTIR Dialect是MLIR的轻量化实现 |
| Dialect定义 | TableGen + ODS(Operation Definition Specification) | 手写C++ + DialectID枚举 | **简化**：远期可引入声明式定义 |
| Dialect变换 | PatternRewriter + DialectConversion框架 | DialectLoweringPass | **简化**：BTIR仅做降级，不做Dialect间变换 |
| 多级IR | 无限层级Dialect嵌套 | 5个固定Dialect(bt_core/cpp/target/debug/meta) | **简化**：5层足够，避免过度复杂 |
| 验证 | ODS自动生成验证逻辑 | DialectCapability + BackendCapability检查 | **对齐** |

**关键决策**：BTIR选择轻量Dialect而非完整MLIR Dialect，理由：
1. BlockType编译器规模不需要MLIR级别的Dialect复杂度
2. 轻量Dialect实现成本远低于完整MLIR集成
3. 5个固定Dialect覆盖BlockType所有需求
4. 远期如需更多Dialect，可渐进扩展DialectID枚举

### 1.8.3 GCC 对标

| 特性 | GCC 实现 | BlockType 对标方案 | 对标程度 |
|------|---------|-------------------|---------|
| 多级IR | GENERIC → GIMPLE → RTL → Machine | bt_core → bt_cpp降级 → 后端内部表示 | **部分对齐**：BTIR层级更简洁 |
| 插件系统 | GCC Plugin API (-fplugin) | CompilerPlugin + PluginManager | **对齐** |
| 诊断系统 | GCC Diagnostics (丰富警告分组) | DiagnosticGroupManager + StructuredDiagnostic | **超越**：增加SARIF/JSON格式 |
| LTO | Full LTO + ThinLTO | 远期：IR缓存 + ThinLTO式增量 | **远期对齐** |
| 缓存 | 无内置缓存 | CompilationCacheManager | **超越** |
| 调试信息 | DWARF5 (GCC 11+) | DWARF5 + CodeView | **对齐+增强** |

### 1.8.4 Rustc 对标

| 特性 | Rustc 实现 | BlockType 对标方案 | 对标程度 |
|------|-----------|-------------------|---------|
| 查询式编译 | 完整查询系统(所有编译计算通过查询) | QueryContext + 基础查询(渐进式) | **部分对齐**：远期完整实现 |
| 增量编译 | 红绿标记 + 依赖图 + 缓存持久化 | 红绿标记算法 + 依赖图追踪 | **对齐**（Phase G） |
| 投影查询 | 函数级增量重编译 | 投影查询模式 | **对齐**（Phase G） |
| 自剖析 | -Zself-profile (查询级性能剖析) | TelemetryCollector + Chrome Trace | **对齐** |
| 错误恢复 | 优雅降级(类型错误后继续编译) | 结构化诊断 + 错误恢复(远期) | **远期对齐** |

### 1.8.5 CompCert 对标

| 特性 | CompCert 实现 | BlockType 对标方案 | 对标程度 |
|------|-------------|-------------------|---------|
| 形式化验证 | Coq证明每Pass语义保持 | PassInvariantChecker + IREquivalenceChecker | **简化对齐**：自动化验证而非Coq证明 |
| 可重现构建 | CompCert本身可重现 | IRModule可重现模式 + SOURCE_DATE_EPOCH | **对齐** |
| 正确性保证 | 数学证明编译正确性 | 差分测试 + 等价性验证 + 不变量检查 | **实用化替代**：无法达到CompCert级别证明，但提供实用验证 |

**关键决策**：BlockType选择实用化验证而非完整形式化验证，理由：
1. CompCert的Coq证明需要专业形式化验证团队，BlockType项目规模不支撑
2. PassInvariantChecker + IREquivalenceChecker + 差分测试可覆盖大部分正确性需求
3. 远期(Phase H)可引入符号执行验证，逐步逼近形式化验证水平

### 1.8.6 Cranelift 对标

| 特性 | Cranelift 实现 | BlockType 对标方案 | 对标程度 |
|------|---------------|-------------------|---------|
| 模块化后端 | 独立指令选择(ISLE) + 寄存器分配 + 代码发射 | InstructionSelector + RegisterAllocator + CodeEmitter | **对齐** |
| 声明式指令选择 | ISLE DSL(完整声明式规则语言) | LoweringRule + .isle规则文件(远期) | **部分对齐**：远期完整实现 |
| 可插拔寄存器分配 | 多种分配策略(Greedy/LinearScan) | RegAllocStrategy(Greedy/Fast/Basic) | **对齐** |
| 安全设计 | Spectre防御 + 可重现构建 | IR沙箱 + 可重现构建 + IR完整性验证 | **对齐+增强** |
| WASM集成 | WASM后端(完整WASM支持) | WASM调用约定 + FFI(远期) | **远期对齐** |
