## 2.3.1 IR→LLVM IR 转换层

### 完整类定义

```cpp
class IRToLLVMConverter {
  ir::IRTypeContext& IRCtx;     // 持有IRCtx便于类型查询（取自方案0）
  llvm::LLVMContext& LLVMCtx;
  const BackendOptions& Opts;
  unique_ptr<llvm::Module> TheModule;

  // 类型映射缓存
  DenseMap<const ir::IRType*, llvm::Type*> TypeMap;
  // 值映射缓存
  DenseMap<const ir::IRValue*, llvm::Value*> ValueMap;
  // 函数映射
  DenseMap<const ir::IRFunction*, llvm::Function*> FunctionMap;
  // 基本块映射
  DenseMap<const ir::IRBasicBlock*, llvm::BasicBlock*> BlockMap;

public:
  /// 构造函数：IRCtx和LLVMCtx必须非空
  /// 约束：LLVMCtx的生命周期必须覆盖整个convert过程
  /// 约束：Opts必须包含有效的TargetTriple
  IRToLLVMConverter(ir::IRTypeContext& IRCtx,
                    llvm::LLVMContext& LLVMCtx,
                    const BackendOptions& Opts = BackendOptions());

  /// 主转换入口：IRModule → llvm::Module
  /// 失败时返回nullptr（不应发生，所有IR指令必须可映射）
  unique_ptr<llvm::Module> convert(ir::IRModule& IRModule);

private:
  /// 类型映射：IRType → llvm::Type（带缓存，同类型仅映射一次）
  llvm::Type* mapType(const ir::IRType* T);

  /// 值映射：IRValue → llvm::Value（带缓存）
  llvm::Value* mapValue(const ir::IRValue* V);

  /// 函数映射：IRFunction → llvm::Function（含参数映射）
  llvm::Function* mapFunction(const ir::IRFunction* F);

  /// 基本块映射：IRBasicBlock → llvm::BasicBlock
  llvm::BasicBlock* mapBasicBlock(const ir::IRBasicBlock* BB);

  /// 函数体转换：逐指令转换IR指令为LLVM IR指令
  void convertFunction(ir::IRFunction& IRF, llvm::Function* LLVMF);

  /// 指令转换：核心switch-case分支（见下方完整分支表）
  void convertInstruction(ir::IRInstruction& I, llvm::IRBuilder<>& Builder);

  /// 全局变量转换：IRGlobalVariable → llvm::GlobalVariable
  void convertGlobalVariable(ir::IRGlobalVariable& IRGV);

  /// 常量转换：IRConstant → llvm::Constant
  llvm::Constant* convertConstant(const ir::IRConstant* C);
};
```

### 后端能力检查集成

```
CompilerInstance::compileFile():
  1. Frontend → IRModule (前端生成全部 IR 特性)
  2. BackendCapability = Backend->getCapability()
  3. RequiredFeatures = IRModule::getRequiredFeatures()
  4. Unsupported = BackendCapability.getUnsupported(RequiredFeatures)
  5. if (Unsupported != 0) → 报告不支持的特性，中止或降级
  6. Backend->emitObject(IRModule)
```

**降级策略**：当后端不支持某些 IR 特性时：
- `ExceptionHandling`：降级为 `call`（无异常表），发出警告
- `DebugInfo`：降级为无调试信息，发出警告
- `DynamicCast`/`VirtualDispatch`：**不可降级**，中止编译
- `VectorOperations`：降级为标量循环，发出警告

## 2.3.5 VTable/RTTI 生成（LLVM 后端专属）

**策略**（取自方案1显式Task）：

VTable 和 RTTI 是 Itanium ABI 特定概念，不属于 IR 层通用语义。在 IR 层中，它们通过**全局常量+函数调用约定**表达。具体的 VTable 布局和 RTTI type_info 结构在 LLVM 后端的 `IRToLLVMConverter` 中生成。

**LLVM 后端 VTable/RTTI 生成流程**：
1. `ASTToIRConverter` 识别 C++ 类需要 VTable/RTTI
2. 在 IR 中创建占位全局变量（`ir::IRGlobalVariable`，类型为 `ir::IROpaqueType`）
3. `IRToLLVMConverter` 在转换时根据 Itanium ABI 填充 VTable/RTTI 的具体布局
4. 虚函数调用在 IR 中为间接调用（通过函数指针），LLVM 后端映射为虚表查找

## 2.3.6 声明式指令选择（ISLE式）

> 来源：09-行业前沿补充优化方案 §2.3

借鉴 Cranelift ISLE（Instruction Selection/Lowering Expressions）架构，引入声明式指令选择机制。

### 指令选择规则（声明式定义）

```cpp
namespace blocktype::backend {

/// 指令选择规则（声明式定义）
struct LoweringRule {
  ir::Opcode SourceOp;           /// 源IR操作码
  ir::DialectID SourceDialect;  /// 源Dialect
  std::string TargetPattern;    /// 目标模式（DSL表达式）
  std::string Condition;        /// 应用条件（可选，如"bitwidth==32"）
  int Priority;                 /// 优先级（多规则匹配时选择最高优先级）

  /// 完整字段定义：
  /// - SourceOp: 必填，指定匹配的IR操作码
  /// - SourceDialect: 必填，默认DialectID::Core
  /// - TargetPattern: 必填，DSL表达式，如 "add32 %x %y"
  /// - Condition: 可选，如 "bitwidth==32 && isSigned==false"
  /// - Priority: 必填，默认1；数值越大优先级越高
  ///   - Priority 1: 通用规则
  ///   - Priority 2+: 优化规则（如常量优化、地址模式）
};

/// 目标指令列表类型定义
using TargetInstructionList = SmallVector<std::unique_ptr<TargetInstruction>, 8>;
/// 语义：一条IR指令可能展开为多条目标指令（如ir::GEP → 多条add+shift）
/// select()返回true表示成功选择，false表示无匹配规则

/// 指令选择器接口
class InstructionSelector {
public:
  virtual ~InstructionSelector() = default;

  /// 为IR指令选择目标指令
  /// 返回值语义：
  ///   true + Output非空 → 成功选择，Output包含目标指令列表
  ///   true + Output空   → IR指令被优化消除（如死代码）
  ///   false             → 无匹配规则，需fallback到默认映射
  virtual bool select(const ir::IRInstruction& I,
                      TargetInstructionList& Output) = 0;

  /// 加载声明式规则文件
  virtual bool loadRules(StringRef RuleFile) = 0;

  /// 验证规则完整性（穷尽性检查）
  /// 返回true表示所有IR Opcode都有至少一条匹配规则
  virtual bool verifyCompleteness() = 0;
};

/// 目标指令（后端内部表示）
class TargetInstruction {
  std::string Mnemonic;         /// 助记符（如 "add rax, rbx"）
  SmallVector<unsigned, 4> UsedRegs;   /// 使用的寄存器
  SmallVector<unsigned, 4> DefRegs;    /// 定义的寄存器
  SmallVector<ir::IRValue*, 2> IROperands; /// 关联的IR操作数
public:
  StringRef getMnemonic() const { return Mnemonic; }
  ArrayRef<unsigned> getUsedRegs() const { return UsedRegs; }
  ArrayRef<unsigned> getDefRegs() const { return DefRegs; }
  ArrayRef<ir::IRValue*> getIROperands() const { return IROperands; }
};

} // namespace blocktype::backend
```

### .isle 规则文件示例

```
;; BTIR → x86_64 指令选择规则
;; 算术指令
(rule (lower (add i32 %x %y)) (add32 %x %y) :priority 1)
(rule (lower (add i64 %x %y)) (add64 %x %y) :priority 1)
(rule (lower (mul i32 %x %y)) (imul32 %x %y) :priority 1)

;; 常量优化
(rule (lower (add i32 %x (const i32 1))) (inc32 %x) :priority 2)
(rule (lower (add i32 %x (const i32 -1))) (dec32 %x) :priority 2)

;; 内存指令
(rule (lower (load i32 %ptr)) (mov32 (mem %ptr)) :priority 1)
(rule (lower (store i32 %val %ptr)) (mov32 (mem %ptr) %val) :priority 1)
```

### 与 IRToLLVMConverter 的关系

```
现有 IRToLLVMConverter
    ↓ 扩展（Phase D 引入 InstructionSelector 接口）
    ├── Phase A-E：IRToLLVMConverter 直接映射（现有方式不变）
    ├── Phase F：引入 InstructionSelector 接口（LLVM后端内部使用）
    │   IRToLLVMConverter::convertInstruction()
    │       ↓ 委托给 InstructionSelector::select()
    │       ↓ 返回 TargetInstructionList
    │       ↓ IRToLLVMConverter 将 TargetInstruction 映射为 LLVM IR
    └── Phase G+：声明式规则文件(.isle)解析器
        InstructionSelector::loadRules("x86_64.isle")
            ↓ 加载声明式规则
            ↓ 编译为高效匹配代码
            ↓ select() 使用规则匹配
```

**兼容性说明**：
- Phase A-E 仍使用 `IRToLLVMConverter` 直接映射，无行为变化
- Phase F 引入 `InstructionSelector` 接口，作为 `IRToLLVMConverter` 的内部子组件
- 声明式规则文件（`.isle`）为远期目标（Phase F+）

## 2.3.7 可插拔寄存器分配

> 来源：09-行业前沿补充优化方案 §2.3

借鉴 Cranelift regalloc2 的可插拔设计，将寄存器分配设计为可插拔接口，后端可选择最适合的分配策略。

```cpp
namespace blocktype::backend {

/// 寄存器分配策略枚举
enum class RegAllocStrategy {
  Greedy,       /// 贪心算法（高质量，LLVM默认）
  Fast,         /// 线性扫描（高速度，调试构建用）
  Basic,        /// 基础算法（简单正确，验证用）
};

/// TargetFunction 类型定义（后端内部表示）
/// 表示一个经过指令选择后的函数，包含目标指令序列
class TargetFunction {
  std::string Name;
  ir::IRFunctionType* Signature;       /// 原始IR函数签名
  std::vector<TargetBasicBlock> Blocks; /// 目标基本块列表
  TargetFrameInfo FrameInfo;           /// 栈帧信息
public:
  StringRef getName() const { return Name; }
  ir::IRFunctionType* getSignature() const { return Signature; }
  std::vector<TargetBasicBlock>& getBlocks() { return Blocks; }
  TargetFrameInfo& getFrameInfo() { return FrameInfo; }
};

/// TargetRegisterInfo 类型定义（目标平台寄存器信息）
class TargetRegisterInfo {
  unsigned NumRegisters;                /// 物理寄存器数量
  SmallVector<unsigned, 16> CalleeSavedRegs; /// 被调用者保存寄存器
  SmallVector<unsigned, 16> CallerSavedRegs; /// 调用者保存寄存器
  DenseMap<unsigned, unsigned> RegClassMap;   /// 寄存器类别映射
public:
  unsigned getNumRegisters() const { return NumRegisters; }
  ArrayRef<unsigned> getCalleeSavedRegs() const { return CalleeSavedRegs; }
  ArrayRef<unsigned> getCallerSavedRegs() const { return CallerSavedRegs; }
  bool isCalleeSaved(unsigned Reg) const;
  bool isCallerSaved(unsigned Reg) const;
};

/// 寄存器分配器接口（可插拔）
class RegisterAllocator {
public:
  virtual ~RegisterAllocator() = default;
  virtual StringRef getName() const = 0;
  virtual RegAllocStrategy getStrategy() const = 0;

  /// 执行寄存器分配
  /// 返回值语义：
  ///   true  → 分配成功，TargetFunction中的虚拟寄存器已替换为物理寄存器
  ///   false → 分配失败（寄存器压力过大），需spill后重试
  virtual bool allocate(TargetFunction& F,
                        const TargetRegisterInfo& TRI) = 0;
};

/// 寄存器分配器工厂
class RegAllocFactory {
public:
  static std::unique_ptr<RegisterAllocator>
  create(RegAllocStrategy Strategy);
};

} // namespace blocktype::backend
```

**策略选择指南**：

| 策略 | 适用场景 | 质量 | 速度 | 实现阶段 |
|------|---------|------|------|---------|
| `Greedy` | 发布构建（-O2/-O3） | 高 | 中 | Phase D |
| `Fast` | 调试构建（-O0/-O1） | 中 | 快 | Phase D |
| `Basic` | 验证/测试 | 基础 | 快 | Phase D |

**兼容性说明**：
- Phase A-E 寄存器分配完全委托 LLVM（现有方式不变）
- Phase D 引入 `RegisterAllocator` 接口和 `RegAllocFactory`
- Phase D 实现 Greedy/Basic 两种策略（Fast为远期）

## 2.3.8 后端管线模块化

> 来源：09-行业前沿补充优化方案 §2.3

借鉴 Cranelift 的细粒度 crate 模块化架构，将后端内部管线拆分为独立子组件，每个子组件有明确的接口和职责。

### 后端管线模块化拆分

```
BackendBase
├── InstructionSelector    (声明式指令选择)
│   ├── 接口：InstructionSelector
│   ├── 输入：ir::IRInstruction
│   └── 输出：TargetInstructionList
│
├── RegisterAllocator      (可插拔寄存器分配)
│   ├── 接口：RegisterAllocator
│   ├── 输入：TargetFunction + TargetRegisterInfo
│   └── 输出：分配后的 TargetFunction
│
├── CodeEmitter            (代码发射)
│   ├── 接口：CodeEmitter
│   ├── 输入：分配后的 TargetFunction
│   └── 输出：目标代码（ELF/Mach-O/COFF）
│
├── FrameLowering          (栈帧降低)
│   ├── 接口：FrameLowering
│   ├── 输入：TargetFunction + ABI信息
│   └── 输出：含栈帧指令的 TargetFunction
│
├── TargetLowering         (目标特定降低)
│   ├── 接口：TargetLowering
│   ├── 输入：ir::IRInstruction（特定Dialect）
│   └── 输出：目标特定的 TargetInstruction
│
└── DebugInfoEmitter       (调试信息发射)
    ├── 接口：DebugInfoEmitter
    ├── 输入：ir::IRModule + 调试元数据
    └── 输出：DWARF5/CodeView调试段
```

### 每个子组件的精确接口签名

```cpp
namespace blocktype::backend {

/// 代码发射器接口
class CodeEmitter {
public:
  virtual ~CodeEmitter() = default;
  /// 将分配寄存器后的TargetFunction发射为目标代码
  /// 输出格式由TargetMachine决定（ELF/Mach-O/COFF）
  virtual bool emit(const TargetFunction& F,
                    const TargetMachine& TM,
                    raw_ostream& OS) = 0;
  /// 发射整个模块（多个TargetFunction + 全局变量 + 调试信息）
  virtual bool emitModule(const SmallVectorImpl<TargetFunction>& Functions,
                           const TargetMachine& TM,
                           raw_ostream& OS) = 0;
};

/// 栈帧降低接口
class FrameLowering {
public:
  virtual ~FrameLowering() = default;
  /// 为函数生成栈帧prologue/epilogue指令
  /// 插入位置：函数入口（prologue）和每个返回点（epilogue）
  virtual void lower(TargetFunction& F,
                      const TargetRegisterInfo& TRI,
                      const ABIInfo& ABI) = 0;
  /// 获取栈帧大小（用于调试信息）
  virtual uint64_t getStackSize(const TargetFunction& F) const = 0;
};

/// 目标特定降低接口
class TargetLowering {
public:
  virtual ~TargetLowering() = default;
  /// 将特定Dialect的IR指令降低为目标特定指令
  /// 例如：bt_cpp的invoke → x86_64的call+landingpad
  virtual bool lower(const ir::IRInstruction& I,
                      TargetInstructionList& Output) = 0;
  /// 查询是否支持某Dialect
  virtual bool supportsDialect(ir::DialectID D) const = 0;
};

} // namespace blocktype::backend
```

### 子组件间的调用序列和数据流

```
BackendBase::emitObject(IRModule)
    │
    ├── 1. InstructionSelector::select(IRInst) → TargetInstructionList
    │       对每个IRFunction的每条IRInstruction执行
    │       数据流：ir::IRInstruction → TargetInstructionList
    │
    ├── 2. TargetLowering::lower(IRInst) → TargetInstructionList
    │       对bt_cpp/bt_target Dialect指令执行
    │       数据流：ir::IRInstruction(Dialect!=Core) → TargetInstructionList
    │
    ├── 3. FrameLowering::lower(TargetFunction, TRI, ABI)
    │       为每个TargetFunction生成prologue/epilogue
    │       数据流：TargetFunction → 修改后的TargetFunction（含栈帧指令）
    │
    ├── 4. RegisterAllocator::allocate(TargetFunction, TRI)
    │       虚拟寄存器 → 物理寄存器（含spill代码）
    │       数据流：TargetFunction(虚拟寄存器) → TargetFunction(物理寄存器)
    │
    ├── 5. DebugInfoEmitter::emitDebugInfo(IRModule, OS)
    │       生成DWARF5/CodeView调试段
    │       数据流：ir::IRModule + ir::debug::IRDebugMetadata → .debug_*段
    │
    └── 6. CodeEmitter::emitModule(Functions, TM, OS)
            发射目标代码文件
            数据流：TargetFunction[] → ELF/Mach-O/COFF
```

**关键约束**：
- 步骤1和2的输出合并为完整的TargetFunction
- 步骤3必须在步骤4之前（栈帧大小影响寄存器分配）
- 步骤5可与步骤6并行（调试信息段独立于代码段）
- 步骤6必须在步骤4之后（需要物理寄存器才能发射）

### 后端管线执行流程

```
IRModule 输入
    ↓
IRToLLVMConverter::convert()（Phase A-E 直接映射）
    或
BackendBase::emitObject()（Phase D+ 模块化管线）
    ├── InstructionSelector::select()     → 选择目标指令
    ├── RegisterAllocator::allocate()     → 分配寄存器
    ├── FrameLowering::lower()            → 生成栈帧代码
    ├── TargetLowering::lower()           → 目标特定降低
    ├── DebugInfoEmitter::emit()          → 生成调试信息
    └── CodeEmitter::emit()               → 发射目标代码
```

**兼容性说明**：
- Phase A-E 后端管线为 `IRToLLVMConverter` 直接映射，无模块化拆分
- Phase D 引入各子组件接口定义
- Phase D 实现后端管线模块化拆分（D-F9 任务）

## 2.3.9 调试信息升级 — 后端调试发射

> 来源：09-行业前沿补充优化方案 §2.12

在IR层建立调试元数据体系，后端通过 `DebugInfoEmitter` 接口将IR调试元数据发射为DWARF5或CodeView格式。

### IR调试元数据定义（引用02）

> **定义引用**：`IRDebugMetadata`、`IRInstructionDebugInfo`、`DebugInfoEmitter`的完整定义见 [02-IR层实现.md](02-IR层实现.md) §2.1.25。本节仅描述后端发射逻辑。

### 后端调试信息发射流程

DWARF5 是当前调试信息标准（GCC 11+默认），相比DWARF4的主要改进：
- 分离调试文件（`.dwo`）支持更快的链接
- 类型单元索引加速调试信息查找
- 增量调试信息支持部分更新

**LLVM后端DWARF5发射流程**：
```
IRModule（含 ir::debug::IRDebugMetadata）
    ↓ IRToLLVMConverter 映射调试元数据
    ↓ ir::debug::DICompileUnit → llvm::DICompileUnit
    ↓ ir::debug::DIType → llvm::DIType
    ↓ ir::debug::DISubprogram → llvm::DISubprogram
    ↓ ir::debug::DILocation → llvm::DILocation
    ↓
LLVM TargetMachine::addPassesToEmitFile()
    ↓ 生成DWARF5调试段（.debug_info, .debug_abbrev, .debug_line 等）
    ↓
目标文件（ELF/Mach-O/COFF）
```

### CodeView 发射说明

CodeView 是 Windows/MSVC 平台的调试格式，用于MSVC调试器兼容：
- Phase D 定义 `DebugInfoEmitter` 接口
- Phase E 实现 CodeView 调试信息发射（E-F8 任务）
- Windows 后端选择 CodeView，其他平台选择 DWARF5

### 编译选项

```
-g                             生成调试信息（DWARF5默认）
-gdwarf-4                      生成DWARF4调试信息
-gdwarf-5                      生成DWARF5调试信息
-gcodeview                     生成CodeView调试信息（Windows）
-gir-debug                     在IR中保留调试元数据
-fdebug-info-for-profiling     生成性能剖析用调试信息
```

## 2.3.11 编译器缓存架构 — 后端缓存集成

> 来源：09-行业前沿补充优化方案 §2.8

引入多层编译缓存架构，后端通过 `CompilationCacheManager` 集成缓存查询和存储，避免重复编译。

### 缓存体系定义（引用02）

> **定义引用**：`CacheKey`、`CacheEntry`、`CacheStorage`、`LocalDiskCache`、`RemoteCache`、`CompilationCacheManager`的完整定义见 [02-IR层实现.md](02-IR层实现.md) §2.1.23。本节仅描述后端集成流程。

### 后端缓存集成流程

```
CompilerInstance::compileFile()
    ↓
1. CacheKey::compute(Source, Options)
    ↓
2. CompilationCacheManager::lookupObject(Key)
    ↓ 命中 → 直接返回缓存的目标文件
    ↓ 未命中 ↓
3. Frontend::compile() → IRModule
    ↓
4. CompilationCacheManager::lookupIR(Key)
    ↓ 命中 → 跳过前端，用缓存IR继续
    ↓ 未命中 ↓
5. CompilationCacheManager::storeIR(Key, IRModule)
    ↓
6. Backend::emitObject(IRModule) → 目标文件
    ↓
7. CompilationCacheManager::storeObject(Key, ObjectData)
    ↓
8. 返回目标文件
```

**集成要点**：
- 缓存查询在编译管线入口处执行，命中则跳过对应阶段
- IR缓存和目标文件缓存独立管理，可分别命中
- `CacheKey` 包含编译器版本哈希，版本升级自动失效
- Phase B 实现 `LocalDiskCache` 基础
- Phase E 实现 `CompilationCacheManager` 完整集成
- 远期（Phase F）实现 `RemoteCache` 远程缓存

### 编译选项

```
--fcache-dir=<path>           指定编译缓存目录
--fcache-size=<size>          最大缓存大小（如 "10G"）
--fcache-remote=<url>         远程缓存端点（远期）
--fcache-no                   禁用编译缓存
--fcache-stats                输出缓存统计
```
