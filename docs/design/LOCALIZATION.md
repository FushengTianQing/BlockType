# BlockType 本地化实现指南

## 概述

本文档提供 BlockType 编译器本地化功能的实现指南，包括翻译流程、测试策略和维护建议。

## 本地化架构

```
┌─────────────────────────────────────────┐
│         源代码（双语混合）               │
└─────────────┬───────────────────────────┘
              │
              ▼
┌─────────────────────────────────────────┐
│         Lexer（词法分析器）              │
│  - 识别中英文关键字                      │
│  - Unicode 标识符支持                    │
│  - NFC 规范化                            │
└─────────────┬───────────────────────────┘
              │ Token 流
              ▼
┌─────────────────────────────────────────┐
│         Parser（语法分析器）             │
│  - 双语语法解析                          │
│  - AST 构建                              │
└─────────────┬───────────────────────────┘
              │ AST
              ▼
┌─────────────────────────────────────────┐
│      Semantic Analysis（语义分析）       │
│  - 双语诊断信息                          │
│  - Unicode 符号表                        │
└─────────────┬───────────────────────────┘
              │ 类型检查后的 AST
              ▼
┌─────────────────────────────────────────┐
│      Code Generation（代码生成）         │
│  - Unicode 名称修饰                      │
│  - 标准兼容目标码                        │
└─────────────────────────────────────────┘
```

## 实现步骤

### Phase 1: Lexer（词法分析器）

**时间：** 3 周

**任务：**

1. **Unicode 标识符识别**
   - 实现 UAX #31 ID_Start 和 ID_Continue 判断
   - 集成 ICU 库进行 Unicode 字符属性查询
   - 实现 UTF-8 编解码

2. **中英文关键字识别**
   - 创建关键字映射表
   - 实现关键字查找（使用完美哈希）
   - Token 类型统一表示

3. **NFC 规范化**
   - 实现 NFC 规范化函数
   - 缓存规范化结果
   - 延迟规范化策略

4. **预处理指令双语支持**
   - 支持中英文预处理指令
   - 统一预处理指令处理流程

### Phase 2: Parser（语法分析器）

**时间：** 4 周

**任务：**

1. **双语语法支持**
   - 修改语法规则支持双语关键字
   - AST 节点记录源语言信息

2. **错误恢复**
   - 双语错误信息
   - 错误位置显示

### Phase 3: Semantic Analysis（语义分析）

**时间：** 6 周

**任务：**

1. **Unicode 符号表**
   - 支持 Unicode 标识符的符号表
   - NFC 规范化的标识符比较

2. **双语诊断信息**
   - 实现诊断引擎
   - 加载诊断消息目录
   - 格式化诊断消息

3. **名称查找**
   - Unicode 标识符名称查找
   - 重载解析支持

### Phase 4: Code Generation（代码生成）

**时间：** 4 周

**任务：**

1. **Unicode 名称修饰**
   - 实现 Unicode 标识符的名称修饰
   - 保证与标准 C++ ABI 兼容

2. **目标码生成**
   - 生成标准兼容的 LLVM IR
   - 兼容性测试

## 翻译流程

### 1. 消息目录维护

```
diagnostics/
├── en-US.yaml    # 英文诊断消息
├── zh-CN.yaml    # 中文诊断消息
└── template.yaml # 消息模板（供翻译参考）
```

### 2. 翻译步骤

1. **提取新消息**
   - 从源代码中提取诊断消息 ID
   - 更新消息模板

2. **翻译消息**
   - 将英文消息翻译为中文
   - 保持参数占位符（`%0`, `%1` 等）
   - 保持消息格式一致

3. **审核翻译**
   - 检查术语一致性
   - 检查语法正确性
   - 检查用户友好性

4. **测试翻译**
   - 编译测试用例
   - 验证诊断输出
   - 修复问题

### 3. 术语表

| 英文术语 | 中文术语 | 说明 |
|---------|---------|------|
| identifier | 标识符 | 变量、函数等名称 |
| token | 标记 | 词法单元 |
| type | 类型 | 数据类型 |
| scope | 作用域 | 名称可见范围 |
| namespace | 命名空间 | 名称空间 |
| template | 模板 | 泛型模板 |
| overload | 重载 | 函数重载 |
| override | 覆盖 | 虚函数覆盖 |

## 测试策略

TEST(UnicodeTest, ChineseIdentifiers) {

## 维护建议

### 1. 消息目录更新

- 新增诊断消息时同步更新所有语言版本
- 定期审核翻译质量
- 保持术语一致性

### 2. 关键字映射更新

- 新增 C++ 标准时同步添加关键字
- 审核中文关键字的自然性和准确性
- 更新文档和测试用例

### 3. 性能监控

- 监控 Unicode 处理的性能影响
- 优化热点代码路径
- 定期进行性能测试

## 常见问题

### Q1: 如何处理 Unicode 标识符的名称冲突？

**A:** 使用 NFC 规范化确保语义相同的标识符有相同的表示。

### Q2: 如何保证生成的目标码与标准 C++ 兼容？

**A:** 使用标准名称修饰规则，Unicode 标识符使用 UTF-8 编码后进行修饰。

### Q3: 如何处理不同编码的源文件？

**A:** 编译器自动检测源文件编码，内部统一转换为 UTF-8 处理。

## 总结

BlockType 的本地化实现遵循以下原则：

1. **标准兼容**：遵循 Unicode 标准（UAX #31）
2. **用户友好**：提供母语编程体验
3. **性能优先**：优化 Unicode 处理性能
4. **易于维护**：清晰的架构和文档

通过这些设计，BlockType 将成为一款真正支持中文编程的现代 C++ 编译器。
