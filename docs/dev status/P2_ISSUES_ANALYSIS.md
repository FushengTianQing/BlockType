### 1. 属性查找统一接口 ⭐⭐ (最容易)

- 影响: 代码维护困难，可能遗漏某些属性

#### 需要的工作
1. **设计统一的属性查询接口** (0.5天)
   ```cpp
   // 新接口
   AttrList getAttributes(const Decl *D);
   bool hasAttribute(const Decl *D, StringRef Name);
   ```

2. **重构现有属性访问代码** (1天)
   - 替换所有分散的属性查找调用
   - 统一从 `Decl::getAttrs()` 获取

3. **添加测试** (0.5天)
   - 验证各种属性的正确获取

#### 依赖关系
- ❌ **无外部依赖**
- ✅ 可以独立执行

#### 风险评估
- **风险**: 低
- **原因**: 只是代码重构，不改变语义

#### 预计工作量

- **文件修改**: ~5个文件
- **代码行数**: +200/-150

### 2. NRVO 分析移至 Sema ⭐⭐ (容易)

#### 当前状态
- 现状: NRVO (Named Return Value Optimization) 分析在 CodeGen 中进行
- 问题: CodeGen 不应该做语义分析，应该在 Sema 阶段完成

#### 需要的工作
1. **在 Sema 中添加 NRVO 分析** (1天)
   ```cpp
   // Sema.h
   struct NRVOCandidate {
     VarDecl *Candidate;
     ReturnStmt *ReturnSite;
   };
   
   llvm::Optional<NRVOCandidate> analyzeNRVO(FunctionDecl *FD);
   ```

2. **将分析结果传递给 CodeGen** (0.5天)
   - 在 `FunctionDecl` 或 `Sema` 中存储 NRVO 信息
   - CodeGen 读取该信息

3. **修改 CodeGen 使用 Sema 的分析结果** (1天)
   - 移除 CodeGen 中的 NRVO 分析逻辑
   - 直接使用 Sema 提供的候选者

4. **测试验证** (0.5天)
   - 确保 NRVO 仍然正确工作

#### 依赖关系
- ❌ **无外部依赖**
- ✅ 可以独立执行

#### 风险评估
- **风险**: 低
- **原因**: 只是职责转移，功能不变

#### 预计工作量
- **总时间**: 3-4天
- **文件修改**: ~4个文件
- **代码行数**: +300/-200

---

### 3. 复数/向量类型支持 ⭐⭐⭐ (中等)

#### 当前状态
- 现状: CodeGen 简化处理复数和向量类型
- 问题: 不支持 `_Complex`, SIMD 向量类型

#### 需要的工作
1. **AST 层支持** (2天)
   - 检查 `ComplexType`, `VectorType` 是否已定义
   - 如果没有，需要添加

2. **类型转换** (2天)
   ```cpp
   // CodeGenTypes.cpp
   llvm::Type *ConvertComplexType(ComplexType *CT);
   llvm::Type *ConvertVectorType(VectorType *VT);
   ```

3. **表达式生成** (3天)
   - `EmitBinaryOperator` 支持复数运算
   - `EmitCastExpr` 支持向量转换
   - 内置函数支持 (如 `__builtin_add`)

4. **常量求值** (1天)
   - CodeGenConstant 支持复数/向量常量

5. **测试** (2天)
   - 复数算术测试
   - 向量操作测试

#### 依赖关系
- ⚠️ **依赖 LLVM IR 的向量/复数类型**
- ✅ LLVM 原生支持，无需额外工作

#### 风险评估
- **风险**: 中
- **原因**: 涉及多个组件，但 LLVM 有良好支持

#### 预计工作量
- **总时间**: 1-2周
- **文件修改**: ~8个文件
- **代码行数**: +800/-100

### 4. RTTI 完整实现 ⭐⭐⭐⭐ (较难)

#### 当前状态
- ✅ **已部分实现**: 
  - `dynamic_cast` 已有基本框架 (`CGCXX.cpp:1638-1743`)
  - `typeid` 已有 typeinfo 生成 (`EmitTypeInfo`)
- ⚠️ **不完整**:
  - `dynamic_cast` 只处理简单情况
  - 缺少完整的层次结构搜索
  - `typeid` 可能不完整

#### 需要的工作
1. **完善 dynamic_cast** (1周)
   - 实现完整的向上/向下/交叉转换
   - 处理虚继承情况
   - 正确处理多重继承

2. **完善 typeid** (3天)
   - 确保 `type_info::name()` 正确
   - 实现 `type_info::before()`
   - 实现 `type_info::operator==`

3. **RTTI 缓存优化** (2天)
   - 避免重复生成 typeinfo
   - 跨翻译单元共享

4. **测试** (3天)
   - 各种继承层次的 dynamic_cast
   - typeid 比较测试

#### 依赖关系
- ⚠️ **依赖虚继承的完整实现** (P2-5)
- ⚠️ **依赖 vtable 的正确性**
- ⚠️ **依赖 libcxxabi 的 __dynamic_cast**

#### 风险评估
- **风险**: 中高
- **原因**: 涉及复杂的运行时类型检查，容易出错

#### 预计工作量
- **总时间**: 2-3周
- **文件修改**: ~5个文件
- **代码行数**: +1000/-200

### 5. 虚继承完整实现 ⭐⭐⭐⭐⭐ (困难)

#### 当前状态
- ✅ **已部分实现**:
  - VTT (Virtual Table Table) 已生成 (`EmitVTT`)
  - Thunk 已生成 (`EmitThunk`)
  - 构造函数中 VTT 使用 (`EmitConstructor`)
  - 虚基类偏移计算 (`GetBaseOffset`)
- ⚠️ **不完整**:
  - 析构函数中的 vptr 恢复可能不完整
  - 虚基类的动态偏移加载可能有bug
  - 钻石继承的测试覆盖不足

#### 需要的工作
1. **完善 vptr 管理** (1周)
   - 确保构造/析构过程中 vptr 正确更新
   - 处理嵌套虚继承

2. **完善 thunk 生成** (1周)
   - 处理复杂的 this 调整
   - 支持返回类型协变

3. **完善虚基类访问** (1周)
   - 运行时偏移加载的正确性
   - 多级虚继承的处理

4. **全面测试** (1周)
   - 钻石继承测试
   - 多层虚继承测试
   - 虚继承+多重继承组合测试

#### 依赖关系
- ❌ **无强依赖**
- ⚠️ 与 RTTI 相互依赖 (dynamic_cast 需要虚继承支持)

#### 风险评估
- **风险**: 高
- **原因**: Itanium ABI 复杂，容易引入微妙 bug

#### 预计工作量
- **总时间**: 3-4周
- **文件修改**: ~6个文件
- **代码行数**: +1500/-300

### 6. 依赖类型表达式 ⭐⭐⭐⭐⭐ (困难)

#### 当前状态
- ⚠️ **未实现**:
  - `CXXDependentScopeMemberExpr` - 依赖作用域成员访问
  - `DependentScopeDeclRefExpr` - 依赖作用域声明引用
- 现状: 这些 AST 节点可能在 Parser/Sema 中创建，但 CodeGen 无法处理

#### 需要的工作
1. **理解依赖类型的语义** (1周)
   - 学习 C++ 模板依赖名称解析
   - 理解两阶段查找

2. **Sema 层完善** (2周)
   - 确保依赖表达式在实例化时正确解析
   - 处理 `typename` 关键字
   - 处理 `template` 消歧义

3. **CodeGen 层实现** (2周)
   ```cpp
   void EmitCXXDependentScopeMemberExpr(CXXDependentScopeMemberExpr *E);
   void EmitDependentScopeDeclRefExpr(DependentScopeDeclRefExpr *E);
   ```
   - 这些应该在模板实例化后被替换，理论上不会到达 CodeGen
   - 但如果到达，需要错误处理

4. **测试** (1周)
   - 复杂的模板依赖测试

#### 依赖关系
- ⚠️ **严重依赖模板系统的完善**
- ⚠️ 依赖 Sema 的两阶段查找正确性
- ⚠️ 依赖模板实例化的完整性

#### 风险评估
- **风险**: 高
- **原因**: 涉及 C++ 最复杂的特性之一

#### 预计工作量
- **总时间**: 4-6周
- **文件修改**: ~10个文件
- **代码行数**: +2000/-500

### 7. 协程完整实现 ⭐⭐⭐⭐⭐⭐ (极难)

#### 当前状态
- ⚠️ **简化实现**:
  - `co_return` - 简化为普通 return
  - `co_yield` - 只求值操作数，无挂起
  - `co_await` - 完全未实现 (`EmitCoawaitExpr` 缺失)
- 现状: 缺少 coroutine frame、promise、suspend/resume 机制

#### 需要的工作
1. **设计 Coroutine Frame 布局** (2周)
   - 研究 Itanium C++ ABI 协程规范
   - 设计 frame 结构 (promise、awaiter、suspend points)

2. **实现 Promise 协议** (2周)
   - `get_return_object()`
   - `initial_suspend()` / `final_suspend()`
   - `return_value()` / `return_void()`
   - `unhandled_exception()`
   - `yield_value()` (for generators)

3. **实现 Awaiter 协议** (2周)
   - `await_ready()`
   - `await_suspend()`
   - `await_resume()`

4. **实现 Coroutine Lowering** (3周)
   - 将协程函数转换为状态机
   - 生成 suspend/resume 入口点
   - 管理协程生命周期

5. **实现 co_await/co_yield/co_return** (2周)
   - 完整的语义实现
   - 异常安全

6. **测试** (2周)
   - 基本协程测试
   - 生成器测试
   - 异步 IO 测试

#### 依赖关系
- ⚠️ **无现有基础** - 需要从头实现
- ⚠️ 依赖 LLVM 的协程 intrinsic (`llvm.coro.*`)
- ⚠️ 可能需要修改 AST 以支持协程变换

#### 风险评估
- **风险**: 极高
- **原因**: 
  - C++ 最复杂的特性
  - 涉及编译器变换
  - 容易引入内存安全问题

#### 预计工作量
- **总时间**: 8-12周 (2-3个月)
- **文件修改**: ~15个文件
- **代码行数**: +5000/-1000

## ⚠️ 关键风险提示

### 高风险项
1. **协程实现** - 可能发现需要重构整个 CodeGen 架构
2. **依赖类型** - 可能需要重写模板实例化逻辑
3. **虚继承** - 微妙的 ABI 细节可能导致难以调试的 bug

### 缓解策略
1. **渐进式开发** - 每个阶段都有可测试的里程碑
2. **充分测试** - 每个特性都要有全面的测试套件
3. **参考 Clang** - 遇到难题时参考 Clang 的实现
