# BlockType 项目 AI 开发指令

## 核心原则：追根溯源，不搞临时方案

### 规则 1：遇到缺失的前置条件时，修复源头而非绕过

当你发现某个功能依赖的前置模块/接口不完整时：

- **正确做法**：追溯该前置模块，补全缺失的接口/实现，然后继续当前任务
- **错误做法**：写 `// TODO`、用临时数据结构绕过、写 workaround 注释、在当前模块内做本应由前置模块做的事

**判断标准**：如果一个缺失项属于前序阶段的职责范围（参考 `docs/PHASE3-GAP-ANALYSIS.md`），就去补全它，不要在当前阶段引入临时方案。

### 规则 2：接口完整性优先于功能完整性

每个模块产出的**接口（头文件中的类/函数声明）必须是完整的**，即使实现是空壳。

- 缺少的方法声明 → 立即补全
- 缺少的 enum 值 → 立即补全
- 缺少的 `classof` 覆盖 → 立即补全
- 缺少的构造函数/工厂方法 → 立即补全

理由：后续阶段的 AI 开发者看到完整接口时，会使用正确的方式；看到残缺接口时，会发明临时的替代方案。

### 规则 3：参照 Clang 的架构模式

本项目架构参考 Clang 编译器。遇到不确定的设计决策时：

1. 查看 `docs/ARCHITECTURE.md` 中的模块职责划分
2. 查看 `docs/PHASE3-GAP-ANALYSIS.md` 中的已知缺口
3. 参照 Clang 的对应实现做决策

### 规则 4：修改前先读文档

在做任何修改之前，先阅读相关文档：
- `docs/ARCHITECTURE.md` - 模块职责和依赖关系
- `docs/ROADMAP.md` - 开发阶段和验收标准
- `docs/PHASE3-GAP-ANALYSIS.md` - Phase 3 的完成度和已知缺口

### 规则 5：禁止的模式

以下模式出现时，立即停止并重新思考：

- `// HACK:` / `// WORKAROUND:` / `// FIXME: temporary` → 不允许
- 在 A 模块中实现本应由 B 模块提供的功能 → 不允许
- 用 `void*` / `intptr_t` / 字符串代替正确的类型 → 不允许
- 复制粘贴已有代码做轻微修改而非抽象公共逻辑 → 不允许
- 跳过测试 "因为前置功能还没实现" → 不允许，先去实现前置功能

## 项目结构

```
include/blocktype/   → 公共头文件（接口契约）
  AST/               → AST 节点定义（跨阶段共享）
  Basic/              → 基础设施
  Lex/                → 词法分析
  Parse/              → 语法分析
  Sema/               → 语义分析
src/                 → 实现
  AST/                → AST 节点实现
  Basic/              → 基础设施实现
  Lex/                → 词法分析实现
  Parse/              → 语法分析实现
  Sema/               → 语义分析实现
docs/                → 文档
  ARCHITECTURE.md     → 架构设计
  ROADMAP.md          → 开发路线图
  PHASE3-GAP-ANALYSIS.md → Phase 3 完成度分析
tests/               → 测试
```

## 模块依赖（严格单向）

```
Basic ← Lex ← Parse ← Sema ← CodeGen
              ↑         ↑
              └── AST ──┘
```

- 上游模块不能依赖下游模块
- AST 是 Parse 和 Sema 的共享层
- 任何反向依赖都是架构错误
